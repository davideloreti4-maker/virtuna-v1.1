# Phase 3 — UI Design Contract

> Visual and interaction contract for the Numen landing **honesty moat + gallery + social proof + conversion**. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope:** fill the four remaining `SectionShell` slots scaffolded heading-only in `src/app/(marketing)/page.tsx` — `#honesty`, `#gallery`, `#proof`, `#cta` — plus an **early proof strip** mounted under the hero / around `#how-it-works` (D-10), so the page **earns belief and converts**. >
> **Out of scope (do NOT build here):** final `.numen-surface` token swap (D-L3 / Phase 4); scroll-reveal choreography (MOT-01 / Phase 4); LCP/OG/a11y launch polish (PERF-01/03 / Phase 4); the final rights-cleared ≥3-niche gallery asset set + real testimonial quotes (D-L4 — Phase 4 / launch; ship structurally complete on placeholders now); use-cases/personas section (post-MVP); double opt-in / confirmation email; named-rival comparison (rejected); app deep-linking.

> **Token posture (carried, locked — DS-01/DS-02/D-L3):** consume the EXISTING `src/app/globals.css` `.numen-surface` scope as placeholder tokens. All color by **bridged token NAME** (`bg-bg`, `text-text`, `bg-panel`, `bg-verdict-good/mixed/bad`, `text-accent`, …) so the Phase-4 hex swap needs zero JSX edits. **NEVER fork or invent tokens. ## Typography

Single typeface (Inter). **Exactly 4 sizes, 2 weights** — identical scale to Phases 1–2 (do NOT introduce a new size or weight).

| Role | Size (mobile → desktop) | Weight | Line Height | Tracking | Tailwind |
|------|------------------------|--------|-------------|----------|----------|
| Section `<h2>` (honesty / gallery / proof / cta) | 30px → 36px | 700 (bold) | 1.2 | -0.02em | `text-3xl md:text-4xl font-bold tracking-tight` (from `SectionShell`) |
| Section lead / subhead | 16px → 18px | 400 (regular) | 1.5 | normal | `text-base md:text-lg leading-relaxed` |
| Body / comparison cell / gallery why-line / testimonial quote / form copy | 16px (why-line 14px→16px) | 400 (regular) | 1.5 | normal | `text-base leading-relaxed` (why-line `text-sm md:text-base`) |
| Verdict band LABEL (gallery card + comparison Numen cell) | 14px → 16px | 700 (bold) | 1.3 | normal | `text-sm md:text-base font-bold` |
| Count figure (proof — the live number) | 30px → 36px | 700 (bold) | 1.2 | -0.02em | `text-3xl md:text-4xl font-bold tracking-tight` (a focal figure, NOT a new display size — reuses the `<h2>` scale) |
| Eyebrow / niche tag / comparison column header / legal-ish microcopy | 14px | 400 (500 optical on `PillChip`/active) | 1.4 | normal | `text-sm text-text-muted` |

**Weights: exactly 2 — regular (400) + bold (700)** (500 only as the existing `PillChip`/button optical-medium; no third display weight). The count figure is the only "big number" on the page — and it is a **real count of real people**, not a verdict and not a hype metric (D-08), so it does NOT violate VOICE Rule 3 (which bans a *verdict* expressed as a number). Headings + the count figure use `text-text`; subheads, why-lines, testimonial body, form helper copy use `text-text-muted`; verdict band LABELs meet **APCA Lc ≥ 60** (plate-backed, see Color). Do NOT apply the reserved serif anywhere.

**Destructive:** none — Phase 3 has no destructive actions (a waitlist signup is additive; a duplicate is treated as success, not an error path with a destructive color).

### Glass / blur posture (Lightning CSS landmine — carried, load-bearing here)

Gallery + testimonial + proof cards are **content on opaque `Surface` plates, NOT glass-over-photo** (REQUIREMENTS forbids glass-over-photo; Lightning CSS strips `backdrop-filter` in prod). The gallery still sits inside/above an opaque `Surface`; the verdict band/label sits on an opaque plate. **No `backdrop-filter` via CSS class anywhere.** If blur is ever unavoidable, apply ONLY via React inline `style={{ backdropFilter: 'blur(Xpx)', WebkitBackdropFilter: 'blur(Xpx)' }}` — recommended: no blur at all this phase.

## Section Contracts

### 1. `#honesty` — kero comparison move (TRUST-01/02 / D-04/D-05)

- **Structure: a true semantic `<table>`** (RESEARCH Pattern 1 — accessibility default for a tabular comparison; a 2-col grid is visually acceptable but the `<table>` gives screen readers the row/column relationship for free).
  - `<caption className="sr-only">Numen compared to virality-score tools</caption>`
  - `<thead>`: `<th scope="col">` (blank/dimension corner) · `<th scope="col">Numen</th>` · `<th scope="col">Virality-score tools</th>`
  - `<tbody>`: each row `<th scope="row">{dimension}</th>` + a Numen `<td>` + a rival `<td>`.
- **Rows (3, Claude's discretion on exact wording — must pass VOICE):** the dimensions contrast *what you get* / *when it's honest* / *what it won't promise* — e.g.
  1. **What you get** — Numen: a `VerdictSwatch good` band + one-line why ("an honest verdict you can act on"). Rival: "a viral score out of 100".
  2. **How it reads** — Numen: "reads your video like your sharpest audience would". Rival: "a black-box percentage".
  3. **What it promises** — Numen: "the one thing to fix — and when it'll land". Rival: "95% accuracy · guaranteed views".
- **D-05 sanctioned-strings rule (HARD):** the rival column is the **ONE place** on the entire page where "viral score", "virality score", "95% accuracy", "% accuracy", "guaranteed views" may appear — used to **label the rejected category**, never as a Numen claim. These strings MUST live ONLY inside `honesty-comparison.tsx` so the page-wide voice gate can scope them (positively assert they're present-and-labelled here, ban-scan them everywhere else — RESEARCH Pitfall 5 / Validation).
- **Visual treatment:** tasteful contrast, **not a teardown**. Both columns on equal-weight `Surface`/`bg-panel` plates with `border-border` hairline cell rules. The Numen column is the calm, confident one (band + why); the rival column states the snake-oil pattern plainly and quietly (`text-text-muted`) — let the contrast do the work, no red "✗ BAD" screaming, no mocking. Optional Lucide `Check` (Numen rows) / `X` (rival rows) as quiet `text-text-muted` cell markers — NOT accent/verdict-colored.
- **Numen cells stay VOICE-clean:** band + why, never a number; no engine jargon.
- Responsive: on mobile the `<table>` may render as stacked row-groups (each dimension = a labelled pair) as long as the semantic `<table>` markup is preserved; on desktop, the 2-data-column table.

### 2. `#gallery` — luma real-Readings gallery (GALLERY-01/02 / D-06/D-07)

- **Layout: a responsive CSS grid** (luma staging, content-dominant). Mobile = single column (`grid-cols-1 gap-6`); desktop = `md:grid-cols-3 gap-8` (NOT masonry — equal-height cards read calmer and pace like kero; masonry adds motion noise). **≥3 cards** across ≥3 distinct niches; **4 cards** is the recommended default (3-col grid + one wrap, or `lg:grid-cols-3` with 3 — planner's discretion, minimum 3 niches).
- **Card anatomy** (each card = a gallery-quality content centerpiece, NOT a feature diagram):
  - `Surface` plate (`bg-panel`, `border-border`, `rounded-[12px]`), `overflow-hidden`.
  - `next/image` real still (720×1280 portrait, matching the Phase-2 `keyframe.webp` spec; vary the existing keyframe across cards if no new clips are supplied — D-07 non-blocking) — **content dominates the card**, generous whitespace, no chrome/fake-window.
  - A `VerdictThrone`-style band + one-line why **per card**, with a **verdict that varies across cards** (good / mixed / bad — honesty by breadth). Label on a plate (APCA Lc ≥ 60).
  - A **niche tag** via `PillChip` (e.g. "Comedy", "Fitness", "Cooking") — quiet `text-text-muted`, `intent="instant"`.
  - Real-shaped, specific one-line why per card (luma specificity; passes VOICE; "only Numen could have written this line").
- **Image discipline:** opaque `Surface` plate, never glass-over-photo (Lightning CSS). Every `<img>` carries a non-empty, factual, in-voice `alt` (see Copywriting).
- **No card hover theater:** at most `hover:bg-panel-2` on an interactive card (carried Raycast/card rule) — NO translate-y, NO border change, NO scroll-reveal. Cards may be purely static (no hover) this phase.

### 3. `#proof` + early strip — social proof (PROOF-01/02 / D-08/D-09/D-10)

- **Primary proof = the live waitlist count** (D-08), read once via cached RPC (`getWaitlistCount`, `unstable_cache` revalidate 60) in `page.tsx` and threaded to BOTH surfaces (one source, two surfaces — D-10). NO second query.
- **Two surfaces:**
  - **Early strip (D-10):** a thin credibility band mounted under the hero / around `#how-it-works` — compact (`py-4 md:py-6`), `border-y border-border` or a single `Surface`, holding the count line + optional niche `PillChip`s. Deliberately NOT a full `py-24` section.
  - **Full `#proof` block:** the section slot lower down — the count line as a focal figure + the testimonial row.
- **D-09 honesty guard (HARD):** show the count number ONLY at/above a threshold; below it, show a **qualitative anchor** — **never render "0 creators"**. Threshold logic lives in the presentational component, not the read.
  - **Threshold = 50** (Claude's-discretion default, RESEARCH recommendation — planner may tune 30–100, surface the chosen value as a one-line decision).
  - **Above threshold:** `{count.toLocaleString()} creators are on the list.` (the count figure typographic treatment above).
  - **Below threshold (qualitative anchor):** a VOICE-clean line that does NOT lie or inflate — e.g. **"Be one of the first creators to read your content honestly."** No number, no fake-precision, no "join thousands".
- **Testimonials (D-08):** build the testimonial card component **structurally complete but placeholder-ready** — NO fabricated quotes (fabricating violates the honesty moat). Card = `Surface` plate + quote (`text-text`) + attribution line (`text-text-muted`, e.g. name · niche). Until real quotes exist (D-L4 / launch), the row renders from an EMPTY/placeholder data array — when empty, the row renders nothing (or a single neutral non-quote credibility line), **never a fake testimonial**. The count is the proof that ships live; testimonials are the slot that fills at launch.
- The count figure is a real count of real people (not a verdict, not a hype metric) — it does NOT violate VOICE Rule 3.

### 4. `#cta` — conversion + real waitlist capture (CTA-02 / D-01/D-02/D-03)

- **A real email waitlist form** (single `email` field → server action → Supabase `waitlist` insert) repeated near the footer in the `#cta` slot; the Phase-2 hero CTA already scroll-anchors here. Both placements feed ONE waitlist, attributed by a hidden `source` field (`landing-hero` vs `landing-footer-cta`).
- **Form layout:** intro copy (h2 already on the slot + a one-line lead) → the form (`max-w-md`). On desktop the email input + submit button may sit inline (`flex gap-2`); on mobile they stack (`flex-col gap-2`). Submit button = `bg-accent text-bg` (carried CTA treatment, matches nav/footer), `min-h-11`, `rounded-lg`.
- **Form primitives:** native `<input type="email">` + a `<button>` styled with the carried accent treatment, driven by `useActionState` + `useFormStatus`. Reuse the existing input/button visual language (token names: `bg-panel`/`bg-bg` field fill, `border-border`, `text-text`, `rounded-lg`, `min-h-11`). No form library.
- **Honeypot:** a visually-hidden text field named `company` — `className="sr-only"`, `tabIndex={-1}`, `autoComplete="off"`, `aria-hidden="true"`. A filled honeypot returns silent success (no insert). It must be invisible to humans AND offscreen for AT.
- **Form states (all required):**

| State | Trigger | Visual / copy |
|-------|---------|---------------|
| idle | initial | email field + enabled submit ("Join the waitlist") |
| submitting | action pending (`useFormStatus`/`pending`) | input `disabled`; button `disabled`, label → "Joining…" (optional Lucide `Loader2` spin, reduced-motion: no spin); no layout shift |
| success | insert ok | swap the form for a calm confirmation: **"You're on the list."** (`role="status"`, `aria-live="polite"`); optional quiet Lucide `Check`. May fade in via `StageBlock` (reduced-motion = static). |
| **duplicate-as-success** | unique-violation `23505` | **identical to success** — "You're on the list." — NEVER reveal the email already exists (no enumeration leak, D-02) |
| error (invalid email) | client `required`/`type=email` + server regex | inline `text-text-muted`/quiet message under the field, `role="alert"`: **"Enter a valid email address."** Field stays editable, value preserved. |
| error (server/network) | non-`23505` insert error | `role="alert"`: **"Something went wrong. Please try again."** Field stays editable. |

- **Focus-visible** ring (accent) on the input + button; the success confirmation receives focus or is announced via `aria-live`.

## Accessibility Baseline (carried + phase-specific)

- **Contrast / APCA:** inherited from `.numen-surface` tokens. Every verdict band LABEL (gallery good/mixed/bad + comparison Numen cell) meets **APCA Lc ≥ 60** via the SOLID `bg-panel`/`border-border` plate (NOT directly on the band, NOT glass-over-photo) — same mitigation `VerdictThrone` already ships. Validate via `scripts/check-apca.ts` if present (Phase-2 demoted the on-band-label check to a composition diagnostic; keep the plate). Body/heading use `text-text`; muted UI uses `text-text-muted` (never an ad-hoc grey).
- **Semantic table (TRUST):** `#honesty` uses a real `<table>` with `<caption class="sr-only">`, `<th scope="col">` + `<th scope="row">` (RESEARCH Pattern 1) — the accessible default for a comparison.
- **Forms:** the email input has a programmatic label (visible or `sr-only` + `aria-label`); the honeypot is `sr-only` + `aria-hidden` + `tabIndex={-1}`; success is `role="status"`/`aria-live="polite"`; errors are `role="alert"`. Submit + input ≥44px (`min-h-11`).
- **Images:** every gallery `<img>` (`next/image`) carries a non-empty, factual, in-voice `alt`.
- **Reduced motion (HARD):** any `StageBlock` reveal (e.g. form success swap, optional gallery appear) degrades to a static opacity show — no translate, no auto-cycle. NO scroll-reveal choreography this phase (Phase 4 / MOT-01). The pending spinner (if used) does not spin under `prefers-reduced-motion: reduce`. No parallax, no presence theater.
- **Single h1:** the hero `<h1>` remains the page's ONLY top-level heading. `#honesty`/`#gallery`/`#proof`/`#cta` each render an `<h2>` (via `SectionShell heading=`); the early proof strip is NOT a section heading (no `<h2>` — it's an inline band). Gallery card niche tags / verdict labels are NOT headings unless wired as `<h3>` under the section `<h2>` (no level skips). The count figure is a styled `<p>`/`<span>`, not a heading.
- **Focus-visible:** every interactive element (email input, submit button, any clickable niche chip/card) gets `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-accent focus-visible:ring-offset-2 focus-visible:ring-offset-bg`.
- **External/anchor behavior:** carried `scroll-mt-20 md:scroll-mt-24` on the slots; smooth-scroll gated behind `prefers-reduced-motion: no-preference`.
- **Verify on a live dev server** (CSS-cache landmine: kill dev server + clear `.next/` when token/CSS changes don't appear). Confirm reduced-motion + form states on a real run.
