<objective>
Build the server-side Ideas pipeline: a new SSE route that authenticates, loads the creator's `creator_profiles` row, assembles the live grounding bundle (`mode:"idea"`, cold-start safe), OVER-GENERATES ~5 idea concepts against the cached `KC_IDEAS_SYSTEM_PROMPT`, extracts each concept's seed hook, runs ONE niche-instantiated Flash pass per candidate (Plan 01's panel param), GATES out sub-floor cards (drop weak, no regen loop — D-03/ENGINE-02), keeps the best 3 (D-13), builds validated `idea-card` blocks embedding the band + lead scroll-quote (Plan 02), attaches the honest grounding line (Plan 02), persists into the user's open thread stamped with KC_GEN_VERSION, and streams content-first (card faces — WITH their lead scroll-quote — first, band chip a beat later — IDEAS-02/D-04). It also adds the `createOpenThreadLazy` helper and the PINNED chain-anchor endpoint for "Develop this →" (D-15/THREAD-05) — the Hooks GENERATION itself is P4.

Resolves the three RESEARCH Open Questions with explicit, RECORDED decisions (see Task notes):
- Q1 seed-hook extraction → structured `json_object` generate with an explicit `seedHook` field (RESEARCH recommendation, with a markered-prose fallback if KC prose craft degrades) — treated as a documented decision with a recorded outcome, NOT a silent mid-flight choice (WARNING-2).
- Q2 latency → parallel SIMs (`Promise.all`) + legible "generating…/scoring…" status events; "content-first" = card-face-before-band-chip.
- Q3 thread → the user's OPEN thread (type:"open", reading_id null); the chain appends to that same thread.

Purpose: the working over-generate→gate→3-cards pipeline (the moat chain begins).
Output: `ideas-runner.ts`, `/api/tools/ideas/route.ts`, `createOpenThreadLazy`, the pinned anchor endpoint.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Ideas API route | Untrusted composer `ask` + chain `anchor` cross into server-side generation; the route is the auth + validation + rate-limit gate (server-side, independent of any client check). |
| service client → threads/messages | The route writes with the service client (RLS bypass); ownership must be enforced via session user_id, never request body (CR-01). |
| model output → persisted blocks | Generated idea-card props are persisted; must pass `validateBlock` at the write boundary. |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-03-08 | Elevation / Info disclosure (IDOR) | open thread / messages | mitigate | `createOpenThreadLazy` + profile load scoped by SESSION user_id (never body); service-client writes ownership-scoped (CR-01). |
