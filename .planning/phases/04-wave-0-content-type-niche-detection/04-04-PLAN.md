<objective>
Gap closure for VERIFICATION GAP-04-01 (severity: blocker). The verifier confirmed Phase 4 SC#2 + SC#5 are broken at production runtime because `content-type-detector.ts:96` calls `fetch(payload.video_url)`, but `normalize.ts:46` populates `video_url` with a Supabase Storage object key (e.g., `"user-abc/video.mp4"`). In production: `fetch()` throws `TypeError: Failed to parse URL`, the D-16 try/catch swallows it, and every video upload silently returns `wave0Result.content_type = null`. The unit tests mask the bug with hardcoded URLs (`wave0-content-type.test.ts:52`); the pipeline integration test only exercises `input_mode="text"` which short-circuits at the detector's input-mode check.

This plan implements fix option (a) from VERIFICATION.md (the narrower diff that mirrors `pipeline.ts:336-352`'s canonical Supabase download pattern), AND **eliminates the foot-gun at the source** (revision applied per plan-checker BLOCKER #1, Option A):

1. **ContentPayload extension** — add `video_storage_path: string | null` so it survives `normalize.ts` (currently dropped after being mapped into `video_url`).
2. **`normalize.ts` (Option A)** — set `video_storage_path` explicitly from `input.video_storage_path` for `video_upload` mode AND set `video_url` ONLY from `input.tiktok_url` (no fallback to `input.video_storage_path`). For `input_mode='video_upload'`, `video_url` is `null`. This kills the alias that caused GAP-04-01: even if the detector logic regressed, there would be no storage key on `video_url` to feed into `fetch()`. The grep audit (performed during revision) confirmed the blast radius is exactly 1 production caller (the detector itself, being fixed) plus 1 bug-encoding test (`normalize.test.ts:143`) plus `videoPayload` in `wave0-content-type.test.ts`. No external consumers of `ContentPayload.video_url` depend on the storage-key alias.
3. **`detectContentType()` signature change** — accept a `SupabaseClient` parameter; when `input_mode === "video_upload"` use `supabase.storage.from("videos").download(payload.video_storage_path)` instead of `fetch(payload.video_url)`. The Gemini Files API upload step (line ~101+) is unchanged — only the buffer-acquisition step swaps.
4. **`runWave0()` signature change** — forward the supabase client to `detectContentType()`. `detectNiche()` does NOT need the client (text-only).
5. **`pipeline.ts`** — pass the existing `supabase = createServiceClient()` (line ~276) into `runWave0()`. No second client instantiation.
6. **Regression-lock tests** — the unit test currently masks the bug by passing `video_url: "https://test-bucket.supabase.co/video.mp4"` (a real URL). The new test uses `video_storage_path: "user-abc/video.mp4"` (a raw storage key, no scheme) AND mocks the Supabase client's storage.download to return a Blob. Plus a new pipeline integration test that exercises `input_mode="video_upload"` end-to-end with a mocked Supabase Blob — this proves SC#2 + SC#5 work in production-shaped data flow. **Plus a normalize.test.ts contract-lock** asserting Option A: video_url is null for video_upload mode and video_storage_path carries the key — re-introducing the alias bug now requires breaking an explicit test.
