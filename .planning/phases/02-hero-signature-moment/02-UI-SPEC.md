# Phase 2 — UI Design Contract

> Visual and interaction contract for the **hero & signature "crowd → score" moment**. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Keystone, highest-craft, highest-risk phase of the milestone.** Bar = one sandcastles-grade signature moment at Linear/Raycast craft, inside the flat-warm Claude-calm system. **Refined > flashy.** Every D-NN below is LOCKED by the human (Davide) in `02-CONTEXT.md` — this spec turns that intent into an executable contract and resolves the four "Claude's Discretion" items on feasibility/perf grounds.

**Accent reserved for (D-02 — the lone reaction accent, explicit allowlist):**
1. The **particle reaction pulse/shift** — dots flash/tint coral *as they react* to the video, then the coalescing stream carries coral inward.
2. The **resolved arc-ring progress stroke** (the filled arc of the gauge).
3. The **score number** at the ring's center.
4. The **primary "Try it free" CTA** fill (existing `shadow-button` primary).
5. **Focus rings** on the CTA / scroll-cue / replay affordance (inherited focus-visible treatment).

**Forbidden:** coral on the subcopy, the H1, the scroll-cue text, the phone frame, the stage border, the ring track, or as any ambient wash/glow. Base particle field is **charcoal/cream only**; coral appears *only* during the reaction and on the resolved instrument. No green/yellow/red score-tier colors (that's the product board's honesty palette, not the marketing hero — the chosen score is a single strong "will-pop" value, so coral alone is honest here).

**Score-honesty constraint:** the canned resolved score MUST be ≥ 70 (the product's `BAND_THRESHOLDS.STRONG`, `verdict-constants.ts`) so that rendering it in coral as a "will pop" outcome is truthful, not a fabricated-green. Lean: **high-80s** (e.g. 87). Exact value is planner/executor's within ≥70.

## Signature Moment — Interaction & Motion Contract

**This section is the heart of the phase.** It binds the locked D-01..D-16 mechanic and resolves the four discretion items. The checker validates against it; the planner turns it into tasks; the executor treats it as the visual source of truth.

### Composition (D-06 / D-07 / D-08 — LOCKED)

- **Centered vertical stack**, top-to-bottom: serif **H1** → Inter **subcopy** → **CTA cluster** (primary "Try it free" + secondary "See how it works ↓") → **large centered signature stage**.
- The stage is a **contained flat-warm scene**: surface = `--color-surface` tone-step, **hairline 6% border** (`--color-border`), **12px radius** (`--radius-lg`). A defined arena for the particles to coalesce within — **NOT** edge-to-edge bleed, **NOT** a full-bleed particle backdrop, **NOT** freeform-floating.
- **The whole composition shares the first viewport on desktop** (the "wow" is immediate, not deferred below the fold). On mobile everything stacks, **headline first**, and the stage may sit just at/below the fold (acceptable — the H1 + CTA must be above it).
- **In-scene source video:** a labelled **`<Placeholder variant="video">` TikTok phone frame** sits inside the stage; the crowd reacts to *it* (D-05). Tells the story your-video → crowd → score. Reuses the Phase-1 slot the human swaps later via the single `src` prop.

### The mechanic (D-01..D-04 — LOCKED)

1. **Crowd unit = a flocking field of hundreds of soft particles**, each dot = one viewer, in **charcoal/cream**. NOT avatar faces, NOT a persona node-cloud. Abstract over literal.
2. **Reaction:** particles **pulse/shift coral** (`#d97757`) as they "react" to the phone-frame video — the lone accent moment.
3. **Resolve = one continuous coalesce:** the crowd **flows inward and coalesces into / hands off to** the score in a single gesture. NOT a separate gauge filling beside a static crowd; NOT a sentiment-sort. The crowd and the number are the same motion (this is the wow).
4. **Score form = arc/ring gauge + centered number**, mirroring the real product's arc-gauge instrument — the coalescing particles land *into* the ring. Reads as "an instrument," not a chart.

### Trigger, rest, fallback, load (D-13..D-16 — LOCKED)

- **Trigger (D-13):** autoplay **once** on hero mount → settle on the resolved score → **stay**. Subtle **replay on hover/click** of the stage. **No continuous loop** competing with the copy/CTA. Calm, "earned" motion.
- **At rest (D-14):** after resolving, the settled field keeps a **very slow, low-amplitude drift/shimmer** (GPU-cheap); the score stays fixed. Reads as a live instrument. **Reduced-motion → frozen** (no drift at all).
- **Fallback (D-15):** the static frame is the **resolved end-state** — phone + settled particle field + arc ring + final score. **One composed still serves all three:** reduced-motion fallback == post-animation rest state == pre-hydration paint. Reduced-motion and pre-hydration visitors get the full payoff immediately, zero motion.
- **Load (D-16):** SSR / first paint renders the **composed still** (no layout shift, no blocking). The client then **lazily mounts** the animated canvas via `dynamic(() => import(...), { ssr: false, loading: () => <ComposedStill/> })` and plays once. The page stays an **RSC**; the moment is a **client island**. (Pattern already proven in `src/components/board/Board.tsx`.)

### RESOLVED — Build approach (discretion item 1) → **custom canvas-2D**

**Decision: hand-rolled HTML5 `<canvas>` 2D, single `requestAnimationFrame` loop. Do NOT pull R3F/three for this, and do NOT add tsparticles / cobe / Magic UI / Aceternity.**

Rationale (the planner commits on this evidence):
- **Bundle:** canvas-2D adds **zero** new dependency weight — it is platform API. R3F+three (~150KB+ gz on top of the island) would bloat the lazy chunk and fight FOUND-06's Lighthouse-mobile-≥90 bar for *no visual gain* — the flat-warm aesthetic is 2D soft dots coalescing in-plane, not a 3D-depth scene. (D-04's "natural home to land in" is a 2D arc.) Pulling a generic particle lib was explicitly vetoed ("avoid heavy generic particle libs that fight the calm").
- **Art-direction control:** the coral-only reaction, the charcoal/cream base, the precise coalesce-into-the-ring choreography, and the exact easing are all far easier to art-direct pixel-by-pixel in raw 2D canvas than through a library's abstraction. Flat-warm tuning stays trivial.
- **Proven-feasible in-repo:** `src/components/hive-demo/hive-demo-canvas.tsx` already runs a DPR-aware, fit-transformed, single-rAF canvas crowd with idle float (and the heavier societies variant ran ~1300 nodes @60fps) — this hero's hundreds of particles are well within proven budget. Reuse its patterns: pre-compute static data, one `getContext("2d")`, `setTransform(dpr,…)`, `getBoundingClientRect` fit, `touchAction: "auto"` (non-blocking scroll), cancel rAF on unmount.
- **SSR-still / client-hydrate split (D-16):** clean — the still is a server-rendered DOM/SVG composition; the canvas only ever mounts client-side inside the `ssr:false` island, so there is no canvas-on-server hazard.

**Score number on coalesce:** drive the center readout with the existing **`useCountUp`** hook (`src/hooks/useCountUp.ts`) — it's on `motion/react` (the light package, not `framer-motion`), already `usePrefersReducedMotion`-gated (it `.jump()`s to the value under reduce). The **arc fill** animates via SVG `strokeDashoffset` (geometry per `ViralScoreRing`, glow/tiers stripped) OR is drawn on the same canvas — executor's call; both are cheap. Keep the resolved ring as **SVG** in the composed still so it is crisp and present pre-hydration without the canvas.

**Library boundary:** the only motion library touched is **`motion/react`** (via `useCountUp` and any Framer entrance on the H1/subcopy/CTA). The particle loop is **vanilla rAF**, gated manually by `usePrefersReducedMotion` (it cannot be reached by the global `<MotionConfig>`). Do not import `framer-motion` directly.

### RESOLVED — Particle count / density, palette steps, easing & timing (discretion item 2)

Committed budget (executor tunes *within* these ceilings — "calm + refined; coral only as the reaction accent"):

- **Particle count:** **desktop ≈ 300–500** dots (CONTEXT's "hundreds"); **never exceed 600** on desktop. This is comfortably under the proven hive-demo budget and keeps the lazy chunk's runtime cost low.
- **Particle palette (charcoal/cream base + coral reaction — discrete steps, no rainbow):**
  - base dots: cream-secondary `#c2bdb4` / cream-muted `#8a857c` at low alpha (0.25–0.55), some charcoal-chip `#2f2e2b` for depth.
  - reacting dots: lerp toward coral-500 `#d97757`; peak alpha ~0.9 at the reaction crest, easing back as they stream inward.
  - **no** white, **no** non-coral hues.
- **Dot size:** soft, **2–4px** rendered radius (DPR-scaled), subtle; favor density over size for the "crowd" read.
- **Timing (total moment ≤ ~3.5s so it never feels like a load-blocker):**
  - settle-in / gather: ~400–700ms
  - reaction wave (coral pulse rippling across the field): ~600–1000ms
  - **coalesce inward → ring + count-up:** ~1000–1400ms (the climax; count-up `useCountUp` `duration` ≈ 1.2s)
  - then **rest** (slow drift begins).
- **Easing:** use the existing `@theme` curves — **`--ease-out-quart`** `cubic-bezier(0.165,0.84,0.44,1)` for the inward coalesce (decisive arrival), **`--ease-out-cubic`** for the reaction wave, and a gentle sine for the at-rest drift. **Avoid `--ease-spring`** on the coalesce (overshoot reads "bouncy/flashy," not calm).
- **At-rest drift amplitude:** ≤ ~1.5px positional, ≤ 0.05 alpha shimmer, period ~6–10s. Barely-perceptible. Frozen under reduced-motion.

### RESOLVED — Mobile sim behavior (discretion item 3) → **default to the resolved still on small screens**

**Decision: on mobile (`useIsMobile`, < 768px) default to the resolved composed still — do NOT run the full particle animation.** Hold the Phase-5 perf bar (FOUND-06, Lighthouse-mobile ≥90).

Rationale & exact rule:
- The composed still already exists for SSR/reduced-motion (D-15), so "still on mobile" is **free** — no extra artifact, no layout shift, identical payoff.
- Mobile GPUs + the perf budget make a 300–500-dot rAF loop the riskiest thing on the page; defaulting it off is the safe, premium choice and matches the CONTEXT lean ("lighter-or-still on mobile").
- **Tiering (use the existing infra, do not reinvent):**
  - **`useIsMobile` true OR `usePrefersReducedMotion` true → render the still only** (no canvas mount). The `dynamic(ssr:false)` island simply isn't mounted; the SSR still stands.
  - **Desktop, motion OK:** mount the island and play once. Wrap with the existing **`detectInitialTier()` / `startFpsSampler()`** (`src/lib/perf-tier.ts`): on `tier === 'low'` or a sustained <40fps drop, **gracefully settle to the still** (cancel the rAF, leave the resolved frame). The animation must degrade *to its own end-state*, never to a blank or a janky loop.
  - Optional (executor's discretion, not required): a **reduced-density** field (~150 dots) on `tier === 'medium'` desktop instead of full off — only if it measurably holds 60fps. When unsure, prefer still.

This makes the still the universal floor: **reduced-motion, pre-hydration, mobile, low-GPU, and FPS-drop all converge on the same crisp resolved frame.**

### RESOLVED — Phone-frame content (discretion item 4) → **static poster placeholder (no looping video for v1)**

**Decision: the in-scene phone uses `<Placeholder variant="video">` as a static labelled poster (no `src` video) for v1.** A looping muted video is **not** added now.

Rationale:
- A real looping clip would need a shipped asset — out of scope (placeholders only; real-asset swap is v2 EXPND-02). The phone is meant to stay a `<Placeholder>` the human swaps later via the one `src` prop.
- A static poster keeps the moment a **canned cinematic about the crowd → score**, not a fake "live demo" (the dishonesty guardrail). The drama lives in the particles + score, not in the phone playing.
- It is the cheapest option for the perf bar and removes an autoplay-video accessibility/bandwidth liability on mobile.
- The `<Placeholder variant="video">` already renders the flat-warm chip + play-triangle overlay (`src/components/marketing/placeholder.tsx`), reading unmistakably as "a TikTok," with **zero layout shift** (aspect-locked). Give it a vertical phone aspect (e.g. `aspect="9/16"`) and a `label` like "Your TikTok".
- **Swap path:** when the human later passes `src="/clip.mp4"`, the same component renders the real muted `playsInline` video in the identical reserved box — the contract already supports it, no rework.

## Accessibility & Performance Contract (binds Phase-5 FOUND-06/07 early)

- **First paint never blocked:** the composed still SSRs; the heavy island is `ssr:false` lazy. No CLS — the stage box is dimension-locked (aspect-ratio) like the `<Placeholder>`.
- **Reduced-motion:** global `<MotionConfig reducedMotion="user">` covers Framer entrances; the **vanilla rAF particle loop is gated manually** by `usePrefersReducedMotion` → renders the still, no drift. (`MotionConfig` cannot reach raw canvas — manual gate is mandatory, per CONTEXT.)
- **Mobile / low-GPU / FPS-drop** all degrade to the same still (see mobile resolution above).
- **Semantics:** H1 is a real `<h1>`; the stage container is `role="img"` with a descriptive `aria-label` (the score is announced); the replay affordance has an accessible name and is keyboard-operable (or is purely an enhancement on a non-interactive still — if replay is mouse-hover-only, it must NOT be the sole way to perceive the moment, since the still already shows the full outcome).
- **Contrast (WCAG AA):** cream-primary on charcoal-app and coral CTA fill with `accent-foreground #1a0f0a` are inherited UAT-passed pairs. The **score number in coral** must clear AA against the ring's interior background (charcoal/surface) — verify ≥ 4.5:1 at build (coral-500 on `#1e1d1b` ≈ passes; if a smaller tick/label sits on coral, use `accent-foreground`).
- **No autoplay audio, no video autoplay** (phone is a static poster v1).
