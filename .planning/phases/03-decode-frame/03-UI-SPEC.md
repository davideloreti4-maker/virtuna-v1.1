# Phase 3 — UI Design Contract: Decode Frame

> Visual and interaction contract for the Decode frame body (Phase 3: DECODE-01/02).
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Scope note:** This is a CONTENT-FILL phase. The Decode frame's geometry, bounds,
> and shell (`DecodeShellNode.tsx`) are already wired by Phase 2. This contract covers
> only what Phase 3 adds inside the hero body: the 4 stacked beat blocks, the 2 stacked
> lanes, the in-flight state, and mobile card-stack parity.

## Spacing Scale

Declared values — 4px base unit, matching BRAND-BIBLE.md §Spacing:

| Token | Value | Decode Frame Usage |
|-------|-------|--------------------|
| xs | 4px | Icon-to-text gap, beat verdict dot spacing; LuckItem internal gap (category tag → note) |
| sm | 8px | Gap between beat label and beat body text |
| md | 16px | Gap between each beat block; gap between beats section and lanes section |
| lg | 24px | Vertical gap between the two lane blocks (repeatable → luck separator) |
| xl | 32px | Outer top padding of the decode body inside `DecodeShellNode` |
| 2xl | 48px | n/a this phase |
| 3xl | 64px | n/a this phase |

Exceptions:
- Beat blocks use `gap-2` (8px) internally between label row and body text (mirrors VerdictNode `SectionHead` mb-3 pattern).
- Lane separator uses `border-t border-white/[0.06]` (1px) + `pt-4` (16px) — not a spacing token, a border rule.
- `DecodeShellNode` outer wrapper inherits `gap-2` from Phase 2 shell (`flex flex-col gap-2`); Phase 3 replaces the single `<p>` child with the full beat+lane structure using `flex flex-col gap-4` for the beat list and `mt-6` before the lanes section.

Both are established Raycast-derived tokens; the 1px difference is intentional in the source components.

## Beat Block Specification (DECODE-01, D-01/D-02)

4 blocks, fixed order, always rendered:

| # | Label (exact string) | Omni source fields | Absent/weak example body |
|---|---------------------|-------------------|--------------------------|
| 1 | `Hook pattern` | `hook_decomposition`, `factors[Scroll-Stop Power].rationale`, `segments[is_hook_zone]` | "The hook is soft — it relies on text overlay alone, with low visual stop power." |
| 2 | `Structure & pacing` | `segments` (count/durations), `video_signals.pacing_score`, `video_signals.transition_quality` | "One continuous shot with no cuts — structure is flat." (n/a — every video has structure; describe honestly) |
| 3 | `The turn` | `segments[].scene_boundary_reason`, `content_summary` mid-point shift, `emotion_arc` inflection | "No distinct turn — it rides one continuous bit." (D-02 canonical example) |
| 4 | `Emotional beat` | `emotion_arc` shape, `factors[Emotional Charge].score`+`.rationale` | "Emotionally flat — it works on novelty, not feeling." |

**Absent/weak visual treatment:** `verdict:'absent'` and `verdict:'weak'` both render the block at `text-white/35` (same component, same size, NOT a different component/icon/badge). The honest body string is always shown. No strikethrough, no warning color, no "N/A" placeholder. The block is muted but present — stable layout per D-02.

**`improvement_tip` field:** Never rendered. Omitted from the decode user message per RESEARCH.md §Decode Prompt Design. If it appears in the data shape, it is silently ignored by the component.

## Lane Specification (DECODE-02, D-03/D-04/D-05)

### Lane 1: "What you can repeat"

- **Header (exact string):** `What you can repeat`
- **Format:** Unordered list — each item is a structural move (1 line ideally, max 2). No category tags.
- **Bullet marker:** Em dash `—` at `text-white/25`, `pl-3` indent. Not a `<ul>/<li>` with browser list-style; use `div` + `::before` or inline prefix span for consistent Raycast styling.
- **Empty state:** Not possible by spec (Qwen returns ≥1 repeatable item). No defensive empty state needed; Zod schema enforces `repeatable.length >= 1`.
- **Voice:** Structural description — "Front-load the payoff before setup" NOT "You should front-load…". Third-person structural moves per D-06/D-07. EXCEPTION: this lane header uses "you" (sanctioned — addresses creator takeaway, per scope note).

### Lane 2: "What was luck / timing"

- **Header (exact string):** `What was luck / timing`
- **Format:** One `LuckItem` per entry. Each item has two stacked lines: a category tag above, a note below. Internal `LuckItem` gap (tag → note): `gap-1` (4px).
- **Category tag strings (fixed taxonomy, D-05):**
  - `timing / trend-moment`
  - `existing audience reach`
  - `algorithmic outlier`
  - `topic / zeitgeist`
- **Note:** 1 short declarative sentence. Third-person. No advice verbs.
- **Visual:** `text-white/40` tag + `text-white/55` note. NOT `text-warning`, NOT `text-error`. Neutral palette — luck is not failure.
- **Empty state:** Not possible (Zod `luck.min(1)` + TS backstop). No defensive empty state needed.
- **Spacing:** `gap-3` (12px) between LuckItems; `gap-1` (4px) inside each LuckItem between category tag and note.

### Visual separation between the two lanes

Single `border-t border-white/[0.06]` with `mt-6 pt-4` above the luck lane header. No heading to group both lanes — each lane header stands on its own. No colored accent or icon to distinguish them — the content itself communicates the distinction.

## In-Flight State (D-09)

Trigger: `isDecoding && !decode` (where `isDecoding = stream.phase === 'analyzing' || phase === 'reconnecting' || phase === 'polling'`).

**Treatment:** Reuse the board's existing pending animation (`motion-safe:animate-skeleton-breathe`) verbatim. Mirror `VerdictSkeleton` from `VerdictNode.tsx`:

```
flex flex-col gap-2
  <span class="text-[10px] uppercase tracking-[0.1em] text-white/45 motion-safe:animate-skeleton-breathe">
    Decoding structure…
  </span>
  <span class="text-[44px] font-semibold leading-none tabular-nums text-white/25 motion-safe:animate-skeleton-breathe">
    —
  </span>
```

**Exact label string:** `Decoding structure…` (lowercase body after capital D, ellipsis, matches board phase message from RESEARCH.md system flow).

**Prohibited:** No fake beat blocks, no fake bullet items, no shimmer bars, no beat-shaped skeletons. The frame shows one animated `—` glyph + the status label. Nothing else.

**Motion-safe:** Animation is gated on `motion-safe:` prefix — respects `prefers-reduced-motion`. When motion is disabled, the elements render at their static opacity with no animation.

## Interaction Contract

Phase 3 is a **read-only display frame**. No user interactions in the Decode frame itself.

| Interaction | Behavior |
|-------------|----------|
| Decode frame renders | No user action needed — auto-runs on remix submit (D-08) |
| In-flight → complete | Frame body replaces in-flight state when `decode != null` — no animation beyond `animate-skeleton-breathe` stopping |
| Hover on beat block | No hover effect — static text blocks. No `cursor-pointer`, no background change |
| Hover on lane item | No hover effect — same |
| Click anywhere in frame | No action — passive display |
| Permalink reload (`/analyze/[id]` for remix row) | Frame reads `permalinkData.variants.remix.decode` directly when `overall_score` is null (m3 marker handling — pitfall m3, RESEARCH.md §Frontend) |

**Accessibility:**
- `aria-busy={isDecoding}` on the `DecodeShellNode` root div (mirrors VerdictNode `aria-busy={isStreaming}`)
- In-flight text is rendered in the DOM (not CSS-only) — screen readers announce "Decoding structure…" while loading
- Beat labels use appropriate semantic HTML (not ARIA roles — `<span>` for inline label, `<p>` for body)
- No `aria-live` region in Phase 3 (Decode has no single dramatic number to announce; the frame fills progressively)
- Luck category tags have no interactive role; they are `<span>` elements, not buttons

## Mobile Card-Stack Parity (Claude's Discretion)

**Context:** `BoardMobile.tsx:37–43` already places `'decode'` at position 2 in `MOBILE_ORDER_REMIX`. The `DecodeShellNode` mounts identically in the `MobileFrameCard` wrapper (DOM, not Konva). Phase 3 fills the body — no mobile-specific layout changes needed.

**Content parity rule:** Beats and lanes render identically on mobile and desktop. Same 4 beats, same 2 lanes, same copy, same typography. No abbreviated mobile variant. The narrow card width (~375px) is accommodated by `max-w-[44ch]` wrapping on body text (existing DecodeShellNode pattern) and `text-wrap: balance` on longer beat bodies.

**Specific mobile sizing:**
- Beat body: `text-[13px]` (same as desktop — readable at mobile)
- No truncation on beat bodies or lane bullets — let text wrap naturally
- Lane bullets: wrap to next line within card width; no overflow hidden

## Error State

If `runDecode` fails after the single Qwen retry (RESEARCH.md §Decode Prompt Design — PATTERN S7):

- The frame renders the error strings from the Copywriting Contract above
- Typography: `text-[13px] leading-[1.4] text-white/45` (subdued, not alarming)
- No retry button in Phase 3 — the decode is the automatic result; a user-facing retry adds complexity deferred to Phase 5 polish
- Sentry captures the failure (PATTERN S7, mirrors `stage11`)
- The frame does NOT collapse or hide — it shows the error message in the body area
