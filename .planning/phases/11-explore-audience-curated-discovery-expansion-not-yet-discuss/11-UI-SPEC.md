# Phase 11 — UI Design Contract: Explore (Audience-Curated Discovery)

> Visual and interaction contract for the Explore in-thread skill. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Design SSOT:** flat-warm **THEME-06** (`src/app/globals.css`) — NOT the legacy Raycast coral/`#07080a` tokens in BRAND-BIBLE's color tables. Where BRAND-BIBLE and the live tokens disagree, **THEME-06 wins** (warm charcoal surfaces, cream text, terracotta-coral `#d97757` accent). BRAND-BIBLE remains authoritative for *patterns* (6% borders, 12px card radius, Inter, card hover = `bg-white/[0.02]`, accent-law).
>
> **Reuse-first mandate:** Explore is NOT a new visual language. Every surface below maps onto existing components (`OutlierTile`, `DiscoverGrid`, `OutlierGridBlock`, `AudienceLens`, the `remix-card` chain, the composer `ComposerControls` popover). This contract specifies only the *extensions* (audience-fit field + on-tap reveal + idle quick-actions + params sheet + serendipity valve) — the planner extends, never reinvents.

1. The tile's **one** primary CTA: **"Remix → Read"** — the single accented element on each tile (existing `OutlierTile`; keep its `#FF7F50` CTA styling — `color:#FF7F50; background:rgba(255,127,80,0.08); border:1px solid rgba(255,127,80,0.30)`).
2. The active **skill pill** terracotta icon + the active row / MAX badge in the composer (existing `ComposerControls`) — when Explore is the active skill.
3. The active state of a **quick-action card** and the params **"Run" / apply** affordance (terracotta `#d97757`, matching the composer popover's active-row treatment `bg-[rgba(217,119,87,0.14)] text-[#d97757] border-[rgba(217,119,87,0.34)]`).

**NOT accent (load-bearing honesty rules):**
- The **audience-fit score** is **DATA, never the action** — it must NEVER be coral/terracotta. It renders in neutral cream + a score-zone tone (green/amber/muted), exactly like the measured multiplier renders neutral. Coral on a tile is reserved for the CTA alone.
- The **measured multiplier** stays neutral `text-foreground` + muted baseline label (existing rule — unchanged).
- Deltas/changes (if any appear) use green/red, never coral.

|-------------|-------|-------|
| (none) | — | **Phase 11 has no destructive actions.** "Track account" is a non-destructive additive write (no delete UI ships this phase — watchlist management/removal is P12). No confirmations required. |

### Fit-score visual encoding (Claude's Discretion D-01 — PRESCRIBED)

**Encoding: a 3-segment horizontal fit bar + a one-word label.** NOT dots, NOT a number.

- **Why a bar, not dots:** the tile already carries a number (the measured multiplier, 20px tabular). A second number (or a 0–10 dot count) reads as a competing score and invites "is this the SIM verdict?" confusion — violating the honesty spine (D-02). A short bar reads as an *estimate gauge*, clearly softer than the hard measured multiplier sitting above it.
- **Why not borrow Sandcastles' borrowed-view badge:** their badge stacks `5.3×` + `114K views` (someone else's past views). Our positioning contrast (the moat tell) is `measured multiplier` + **`fit · your audience, predicted`** — so the fit row must read as *yours, forward-looking*, visually distinct from the measured-views row.

**Spec:**
- A thin bar (`h-1.5`, `rounded-full`, full tile width) on the elevated surface track (`bg-white/[0.06]`), filled to one of 3 quantized levels — **Strong / Fair / Weak fit** (mirrors the band vocabulary already in the system; do NOT introduce a 0–100 number).
- Fill tone by level: Strong → `--color-success`; Fair → `--color-warning`; Weak → `--color-foreground-muted` (muted, not red — a weak *fit estimate* is honest under-promise, not an error). NEVER coral.
- A label beneath: `text-[11px] font-semibold uppercase tracking-wide text-foreground-muted` reading **"FIT · {Strong|Fair|Weak}"** with a quiet, always-present provenance word **"predicted"** in the muted sub-line (e.g. `"Fit for your audience · predicted"`).
- Placement: directly **below** the measured-multiplier badge row, **above** the VideoCard metrics grid — so the eye reads measured-fact → predicted-fit → metrics → CTA, top to bottom.
- **Degrade gracefully (D-02):** when no calibrated audience is active (General/null) or no fit signal is available, **omit the fit bar+label entirely** and show only the measured multiplier (exactly today's `OutlierTile`). Never render an empty/zero bar, never fabricate a level.
- **Honesty hard-bind (D-02):** the fit bar is a re-ranked-math estimate. It carries NO persona quote, NO verdict, NO first-person voice. Real persona reactions appear ONLY on tap (§On-tap reveal).

### On-tap reaction reveal (D-04 — the moat chain)

Tapping the tile's **"Remix → Read"** CTA is the committed action (the eager fit bar already served the cheap glance — no redundant cheap diagnostic, per D-04). On tap:
1. Launch the existing `discover→remix` chain handoff (CHAIN_HANDOFFS — already registered; reuse, do not fork). The CTA enters its pending state (existing `remixPending` → label "Remixing…").
2. The chain generates the creator's **own version** (Remix), SIM-tests it, and lands on a **Read** in-thread — reusing the `remix-card` block + the `reading/` renderers verbatim. No new Read layout.
3. The **real persona reaction** (quotes + stop/scroll verdict) is rendered by the **P9 reaction primitive only here**, from real SIM output, via `cardScrollQuoteReactions` + `AudienceLens` — never fabricated, anchored to the lead persona's real verbatim, others quote-empty. This is the first point a persona *voice* appears in the Explore flow.
4. The transition follows the established content-first SSE pattern (progress checklist → card face → band chip), matching `HooksThreadView` / the remix runner — Explore does not invent its own loading choreography.

## Surface-by-Surface Contract

### 1. Explore result tile (`OutlierTile` EXTENDED — EXPLORE-02/03)

Top-to-bottom read order:
1. **Measured-multiplier badge row** (existing, unchanged) — `{n}×` 20px/600 tabular neutral + `vs own`/`vs niche` muted label, with the source sub-tag chip on the right.
2. **NEW: audience-fit row** — the 3-segment fit bar + "FIT · {level}" label + "Fit for your audience · predicted" sub-line (see §Color → Fit-score encoding). Omitted entirely on General/no-signal.
3. **VideoCard metrics grid** (existing, reused verbatim — thumbnail/caption + 4-col Eye/Heart/Comment/Share).
4. **NEW: "+ Track account" affordance** — a quiet borderless text-button (cream-muted, `text-xs font-medium`), left-aligned above or inline-left of the CTA; toggles to "Tracking ✓" on write. Non-accent (it's secondary to Remix).
5. **Primary CTA "Remix → Read"** (existing, unchanged) — the single accent element.

**Planner note (D-04 discretion):** prefer **extending `OutlierGridBlockSchema`** with an optional `fit` field (`{ level: "Strong"|"Fair"|"Weak" } | null` — nullable/optional so existing persisted outlier-grid blocks stay valid, no migration, mirroring the `predictedFailureMode` precedent) + an optional `trackable: boolean`, over adding a parallel `explore-grid` block. Keep the honesty contract: the block carries NO `band`/`model`/numeric score for the measured tile; the `fit.level` is the only added signal and is explicitly an estimate, not SIM output.

### 2. Tile → Remix → Read flow (D-04)

See §Color → On-tap reaction reveal. Reuses `discover→remix` CHAIN_HANDOFFS → `remix-card` block → `reading/` renderers → `AudienceLens` (real reaction). No new layout, no new loading choreography. The reveal of a persona *voice* happens only here.

### 3. Explore idle / start-screen (D-06/D-07 — EXPLORE-04)

Explore's thread view **owns its own idle state** (precedent: `ChatThreadView` / `HooksThreadView` `isIdle`/`isEmpty` ownership). Layout in the `max-w-[760px] mx-auto … gap-6 px-4 py-6` thread column:
- Heading (16px/600) + body (14px/400 secondary-cream) — copy verbatim above.
- **2–4 quick-action cards** (PRESCRIBED count: **3** — niche / competitors / serendipity, covering the audience-derived + watchlist + serendipity-valve trio named in scope). Each card: transparent bg, 6% border, 12px radius, 20px padding, title (16px/600) + sub-line (14px/400 muted), a phosphor line-icon (compass/people/sparkle). Hover `bg-white/[0.02]` only (no lift). Tap runs a preset pull. Active/pressed = terracotta border+tint.
- Cards stack 1-up on mobile (≤640px), 2–3-up wider via the same `auto-fill` minmax pattern.
- **Degrade:** if no tracked accounts exist, card 2 ("What competitors shipped") shows a quiet "Track an account first" disabled sub-state rather than firing an empty pull (honesty — never a fabricated competitor feed).

### 4. Params sheet (D-06 — EXPLORE-01)

**PRESCRIBED form factor: an upward popover beside the composer skill pill** (NOT inline, NOT a bottom sheet) — matching the established `ComposerControls` `Popover` shell exactly (`bottom-[calc(100%+12px)]`, `bg-[#211f1d]`, 6% border, 12px radius, `max-h-[60vh] overflow-y-auto`, popover-everywhere on desktop AND mobile). Triggered by a "Search" control sitting in the composer's left cluster, beside the audience picker.

Contents (top to bottom): Niche/keywords input · Accounts input · Time-window segmented control (Today/Week/Month) · **Serendipity slider** (the widen-beyond-niche valve: a labeled range from "On-niche" to "Surprise me", terracotta fill on the active track) · **"Run Explore"** apply button (terracotta, the popover's one accent). Inputs reuse THEME-06 input styling (42px height, `rgba(255,255,255,0.05)` bg, 8px radius). Labels 14px/400, helper 11px muted.

### 5. Loading state (the pull)

Reuse `DiscoverGrid`'s `state="loading"` tile-skeleton grid verbatim (Skeleton shimmer, no fake %). Extend only the lead line copy to name the scoring step ("Pulling outliers and scoring them for your audience…"). Tile-skeleton footprint already matches the tile (badge row + card + CTA); add a thin skeleton bar between the badge row and the metrics grid to reserve the fit-bar slot so cards don't reflow when fit resolves.
