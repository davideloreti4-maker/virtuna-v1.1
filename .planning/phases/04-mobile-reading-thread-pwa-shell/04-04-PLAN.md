<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client ↔ auth-gated API | `GET /api/analysis/[id]` returns 401 without a session and is per-user (RLS + user_id) |
| browser ↔ service-worker cache | the SW intercepts navigations + the analysis GET and stores responses on-device |
| npm registry → install | a new third-party dependency (Serwist) enters the build |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-CACHE | Information Disclosure | SW caching of `GET /api/analysis/[id]` (auth-gated, per-user) | mitigate | NetworkFirst (network authoritative, cache = offline fallback) scoped to `/api/analysis/`; NEVER in `precacheEntries`; never cache-first. A precache or cache-first strategy here is a cross-user info-disclosure bug (Pitfall 4). Logout cache-clear hook tracked for Plan 03/Phase 5 (NetworkFirst lowers risk meanwhile). |
| T-04-SCOPE | Tampering | service-worker scope intercepting unintended routes | mitigate | SW scope = `/`; `POST /api/analyze` never cached (online-only, D-05) — asserted by grep that analyze is absent from sw.ts caching |
| T-04-STALE | Information Disclosure | stale Reading served offline after access revoked | accept | NetworkFirst prefers live network; only an already-viewed Reading caches; bounded by the cache name. Residual risk low; full per-session partitioning deferred. |
| T-04-SC | Tampering (supply chain) | `@serwist/next` / `serwist` install | mitigate | slopcheck unavailable → blocking-human checkpoint verifies `npm view` + npmjs.com BEFORE install; pinned to `9.5.11` (no preview `10.0.0-*`); both have no postinstall (RESEARCH audit) |
</threat_model>
