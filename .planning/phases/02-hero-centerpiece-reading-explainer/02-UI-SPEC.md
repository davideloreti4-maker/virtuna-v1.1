# Phase 2 — UI Design Contract

> Visual and interaction contract for the Numen landing **hero centerpiece + Reading explainer**. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope:** fill the existing `#hero` and `#how-it-works` SectionShell slots (scaffolded empty in Phase 1). The hero = a real Reading staged on a real creator video, full-bleed, chrome-minimal (krea/luma content-as-hero), with the verdict shown as a calibrated band + one-line why anchored ON the content (the "throne"), revealed via `StageBlock` / `numen-ease-calm` as a calm auto-reveal **loop** with a `prefers-reduced-motion` static fallback. Below it, a static 3-step explainer (upload → engine reads → verdict + why) reusing the SAME real Reading artifacts. Primary hero CTA present, scroll-routing to `#cta`. Covers **HERO-01..04, READ-01, READ-02, CTA-01**.
>
> **Out of scope (do NOT build here):** honesty/comparison section, gallery, social proof, conversion UI + live waitlist capture, footer-CTA repeat (Phase 3); final token swap, scroll-reveal choreography MOT-01, LCP-optimized hero media, OG art, full a11y pass, palette direction D-L1 (Phase 4). Live interactive Reading, rotating multi-niche verdicts, interactive step-scrubbing — all deferred.
>
> **Token posture (carried, locked):** consume the EXISTING `src/app/globals.css` `.numen-surface` scope as placeholder tokens (DS-01/DS-02/D-09 + Phase-2 D-09 carry). All color by **bridged token NAME** (`bg-bg`, `text-text`, `bg-verdict-good`, …) so the Phase-4 hex swap needs zero JSX edits. **NEVER fork or invent tokens.** **NEVER interpolate** verdict classes (`bg-${verdict}`) — Tailwind can't see dynamic strings; use literal strings via `tailwind-variants` (the `VerdictSwatch` pattern).

**Verdict throne color contract (HERO-03 / D-03 / D-04):**

### Glass / blur posture (Lightning CSS landmine — carried)

Recommended: no blur in the hero.

## Hero Interaction & Motion Contract (HERO-04 / D-01 / D-13)

> The single most load-bearing interaction of the phase. Driven entirely by the existing `StageBlock` — no new motion primitive.

- **Centerpiece = a recorded / pre-captured stage-reveal LOOP** (D-01, resolves D-L2). NOT live-interactive, NOT a single static appear. One real captured Reading is replayed full-bleed via `StageBlock` as a **calm auto-reveal loop**, chrome-minimal.
- **Loop mechanics:** drive a controlled `show` state across the captured stages so each stage block materializes in sequence via `StageBlock` (`opacity` calm tween ~400ms + high-damping translate, no overshoot), dwells on the completed verdict throne, then resets and replays. Loop cadence, dwell-on-verdict duration, and whether it pauses on the verdict before restarting are **Claude's discretion** — but MUST stay **calm**: no bounce, no presence theater, no flashing, no rapid cycling. Reference dwell ≥ a few seconds on the verdict so a visitor reads the band + why before the loop restarts.
- **Reduced-motion fallback (HARD — HERO-04 / D-14):** under `prefers-reduced-motion: reduce`, the loop MUST degrade to a **static final state** — the fully-revealed Reading + verdict throne shown at rest (opacity 1, no translate, no looping/auto-advance). `StageBlock` already zeroes translate + duration under reduced motion; the loop CONTROLLER must additionally NOT auto-cycle (render the completed end-state directly). Verify on a real reduced-motion check.
- **No auto-playing audio.** If the captured bundle is a short muted video (one of the allowed asset formats), it MUST be `muted` + `playsInline` + `loop`, and STILL honor reduced-motion by showing a static poster frame instead of playing (HERO-04).
- **LCP note (Phase-2-aware, full optimization is Phase 4):** the artifact is the largest above-fold element. The D-13 spike measures its weight/LCP and gates the asset path (primary live-capture export vs staged-real-keyframe fallback, D-02). Ship a poster/first-frame that paints fast; the loop may hydrate after. Heavy LCP tuning (responsive sources, preloading) is Phase 4 — but do not ship an artifact that blocks first paint.

**Asset authenticity (HERO-02 — hard):** the artifact is a **real Reading on a real creator video** — real keyframe(s) + real-shaped verdict + real why. **FORBIDDEN:** stock photography, a fake browser/window chrome, a fabricated/abstract diagram standing in for the Reading, glass-over-photo. Either capture path (D-02 primary export, or staged-real-keyframe fallback) satisfies this; both forbid the above.

## 3-Step Explainer Contract (READ-01 / READ-02 / D-05 / D-06)

- **Static 3 steps** (kero 3-feature-card adaptation) in `#how-it-works`, demonstrating the real flow: **upload → engine reads → verdict + why**.
- **Each step shows the SAME real Reading content** (READ-02, hard): step 1 "upload" = the real creator keyframe; step 2 "reads" = a real stage-read excerpt from the captured Reading; step 3 "verdict" = the real `bg-verdict-good` band + one-line why. Each step is a real-content demonstration, **NOT an abstract diagram or icon-only card**.
- **Layout:** mobile = single stacked column (3 cards, `gap-6`); desktop = 3-column row (`md:grid-cols-3 gap-8`), equal-height cards on `bg-panel` + `border-border` + `rounded-[12px]` (carried card radius), inner `p-4 md:p-6`.
- **"Content is both demo and navigation" (READ-02) — honored LIGHTLY (D-06):** steps are anchored and **MAY be clickable to scroll to / focus the hero** (optional enhancement — Claude's discretion whether to ship clickable-to-hero or purely static this phase). If clickable, each step is a real interactive element with a visible `focus-visible` ring and ≥44px hit target. **Full interactive scrubbing** (a step driving/scrubbing the hero loop) is **DEFERRED** — do NOT build it.
- Card hover (only if interactive): `bg-white/[0.02]` equivalent via token (`hover:bg-panel-2`) — NO translate-y, NO border change (carried Raycast/card rule).

## Accessibility Baseline (carried + phase-specific)

- **Contrast / APCA:** inherited from `.numen-surface` tokens (APCA-gated upstream). The verdict band LABEL on `bg-verdict-good` MUST meet **APCA Lc ≥ 60** (the `VerdictSwatch` gate — validate via `scripts/check-apca.ts` if present). Do NOT introduce off-token text colors over the keyframe; if the photo hurts legibility use a solid `bg-panel` plate behind the band, not an ad-hoc color.
- **Reduced motion (HARD, HERO-04):** the hero loop degrades to a static fully-revealed end-state (no auto-cycle, no translate) under `prefers-reduced-motion: reduce`. No parallax, no auto-playing presence theater. Muted-video assets show a static poster under reduced motion.
- **Single h1:** the hero `<h1>` remains the page's ONLY top-level heading (D-10). `#how-it-works` is an `<h2>`. Step titles are NOT headings unless wired as `<h3>` under the `<h2>` (no skips).
- **Focus-visible:** any interactive element (hero CTA, optional clickable steps) gets a visible ring: `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-accent focus-visible:ring-offset-2 focus-visible:ring-offset-bg`.
- **Hit targets:** CTA + any clickable step ≥44×44px (`min-h-11`).
- **Hero CTA behavior (CTA-01 / D-07):** the hero CTA is a real, functional control that **scroll-routes to the `#cta` anchor** (the slot exists from Phase 1). Honor `scroll-margin-top` (carried `scroll-mt-20 md:scroll-mt-24`) and gate smooth-scroll behind `prefers-reduced-motion: no-preference`. **Live waitlist capture is DEFERRED to Phase 3** — do NOT build email capture here. Do NOT deep-link into the app `/analyze` (may be unreachable from the landing deploy).
- **Verify on a live dev server** (CSS-cache landmine: kill dev server + clear `.next/` when token/CSS changes don't appear).
