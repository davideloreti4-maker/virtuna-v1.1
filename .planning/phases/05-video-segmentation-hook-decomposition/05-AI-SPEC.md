## 1. System Classification

**System Type:** Extraction — multi-modal structured data extraction (video → 3 typed JSON outputs)

**Description:**
A pipeline stage inside the Virtuna prediction engine that takes a TikTok-style short-form video (single Gemini Files API upload) and runs three parallel scoped `generateContent` calls — hook (0-5s, Gemini 3 Pro), body (5s → end-3s, Gemini 3 Flash), CTA (last 3s, Gemini 3 Flash) — producing a widened `GeminiVideoAnalysis` shape with hook decomposition (6 scores), body video_signals (4 signals), and CTA segment (presence-aware). Used by the Wave 1 prediction pipeline and consumed by the aggregator + result-card UI (M2). "Good" means: every segment returns a Zod-valid structured output grounded to the actual video content, the three calls run in parallel against a single shared upload, partial failures degrade gracefully (1-of-3 fail → ship partial result + warning), and per-segment cost stays under the ~2.0¢/video budget.

**Critical Failure Modes:**
1. **Schema drift / parse failure** — Gemini returns JSON that violates the Zod schema for any segment, breaking the merge and corrupting downstream aggregator math.
2. **Hallucinated hook decomposition** — Pro hook call invents scores ungrounded to video content (e.g., text_overlay_score=8 when no on-screen text exists, or weakest_modality named for a modality the video doesn't contain).
3. **Forced-CTA false positive** — CTA segment scores `cta_present=true` with non-null `strength` for videos that have no actual call-to-action, polluting the content-type-aware penalty rule (D-06) and pushing tutorial/B-roll content into the wrong reward bucket.
4. **Silent partial-failure masking** — 1 or 2 segments fail but the merged result and SignalAvailability flags don't reflect it, so the aggregator weights phantom signals and the user sees a confidently-wrong score.
5. **Cost runaway on Pro segment** — Hook prompt drifts longer than design (rubric + niche + creator context), pushing per-segment cost above the ~1.3¢ ceiling and breaking the milestone ≤$0.075/video budget cap.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Social media / short-form video virality prediction (TikTok-primary, IG Reels + YouTube Shorts adjacent). The system is a **pre-publish creative coach** — it scores a draft vertical video and tells the creator what is working and what to re-shoot, *before* the algorithm gets a vote.

**User Population:** Paid Virtuna users — solo content creators (nano- to mid-tier, 1K to 500K followers) and small in-house brand/agency teams running paid social. They upload a near-final TikTok/Reels/Short and want (a) a 0-100 virality score, (b) a per-modality breakdown of the hook, and (c) one actionable change for the weakest piece. Sophistication ranges from "this is my first TikTok" to "I've been growing for 18 months and want a second opinion on my hook." The engine output also feeds the ML calibration training set (Phase 10) as a labeled-feature source, so a second consumer is the prediction model itself.

**Stakes Level:** Medium-High. No physical, financial, regulatory, or safety harm — but trust harm is severe and one-way. A false-positive *"viral"* score that scoreboards a weak video is more retention-damaging than a false-negative *"weak"* score, because the creator posts the bad video, it flops, and they blame the tool. A false-negative is recoverable (creator re-shoots, then posts and wins). A false-positive burns the relationship. The output is also a feature in a downstream ML model — silently confident wrong scores propagate into prediction calibration and compound across users.

**Output Consequence:** Drives the per-modality breakdown surfaced in the M2 result-card UI (creator-facing) AND feeds the Phase 10 aggregator + Platt calibration training set (model-facing). Specifically: `hook_decomposition.weakest_modality` becomes the headline "fix this" guidance the creator sees; `cta_segment.cta_present` × `wave0Result.content_type.type` triggers the aggregator's content-type-aware CTA penalty (D-06); the four hook sub-scores feed the ML re-train as features. Wrong scores propagate downstream into both **creator behaviour** (which clips they re-shoot, which they post) and the **prediction model** (which features the model learns to weight). Both consumers need the scores to be grounded in what is actually on screen — not hallucinated against absent modalities.

#### Niche-aware rubric variation (light overlay, not rewrite)

The same dimensions apply across all 5 corpus niches (Beauty, Fitness, Education, Comedy, Lifestyle), but **what scores well differs**. The eval rubric and the LLM-judge prompt should carry these biases explicitly so a niche-blind score doesn't get applied:

- **Beauty** — rewards aesthetic-first hooks: `visual_stop_power` weighted heavily (the visual IS the product), close-up product/face shots score well; soft-spoken first words are fine.
- **Fitness** — rewards transformation-promise hooks: opener that promises a body/strength outcome (`first_words_speech_score`), high-energy audio (`audio_hook_quality` weighted up), body shots OK in safe-zone.
- **Education** — rewards curiosity-gap question hooks: question-form openers ("Did you know..."), `text_overlay_score` weighted up (educators lean on text), low-cognitive-load preferred (one concept per video).
- **Comedy** — rewards setup-payoff timing: visual stop-power can be unusual (B-roll absurdism OK), `cta_present=false` is *correct* (comedy with hard CTA often dies), pacing tighter than other niches.
- **Lifestyle** — rewards aspirational-frame hooks: cinematic visuals (`visual_production_quality` weighted up), mood-music coherence critical, CTA optional. *Source for niche-specific frames:* [Influencer Advisory "How to Grow on TikTok 2026"](https://influenceradvisory.com/blog/how-to-grow-on-tiktok/), [Supliful Top TikTok Niches 2025](https://supliful.com/blog/tiktok-niches), [Exercise.com Fitness Niches](https://www.exercise.com/grow/best-tiktok-fitness-niches/).

### Known Failure Modes in This Domain

> Domain-specific, not generic LLM hallucination. These map directly to Section 1 Critical Failure Modes #2 and #3 and inform the reference-dataset stratification in Section 5.

1. **B-roll establishing shot mistaken for the "hook frame."** Many videos open with a 0.3-0.5s wide-shot or product-on-background before cutting to the actual hook. The model can latch onto the establishing shot and score `visual_stop_power` against IT rather than against the post-cut frame that's doing the real work. Manifests as: low `visual_stop_power` on videos that growth coaches would unanimously rate high. *Mitigation cue:* eval dataset must include videos with a deliberate cold-frame intro and the labeled `visual_stop_power` should reflect the post-cut frame.
2. **Brand-watermark scored as content `text_overlay_score`.** Creator's logo / channel name / corner watermark is on-screen text but is NOT content text. Model can score a strong `text_overlay_score` because text exists, when the text is just brand chrome. Manifests as: tutorials with zero content overlay scoring 7+/10 because of a corner watermark. *Mitigation cue:* prompt must explicitly distinguish "creator-brand text/watermark" from "content text," and dataset labels must show the difference.
3. **Misidentified `weakest_modality` against absent modalities.** Pure-visual hooks (no spoken words, ambient music only) can produce low `audio_hook_quality` and `first_words_speech_score` *as artifacts of absence*, not weakness. If the model names one of those as `weakest_modality`, the creator-facing actionable becomes "improve your speech" on a video that deliberately has no speech. Manifests as: creator says "but there's no speech in my video" and trust evaporates. *Mitigation cue:* the schema's `rationale` field on the hook segment must explicitly call out absent modalities; the `weakest_modality` choice must be conditional on presence.
4. **"Trend participation" hook scored as low originality.** Matching a viral format (using the trending sound, doing the dance, recreating the meme template) IS the strategy for trend-jumpers within the 72h peak window. Model can score these as derivative / unoriginal when the entire point is fidelity to the trend. Manifests as: trend-correctly-executed video scores 5/10 because the model "saw it before." *Mitigation cue:* model should NOT weight originality directly on this rubric; trend-fit detection is Phase 8's job (benchmark retrieval) — Phase 5 should score what's on screen.
5. **CTA confusion: TikTok UI vs creator CTA.** TikTok overlays a native "Follow" button on every video and Reels has a "Like / Comment / Send" rail. These are platform UI, not creator CTAs. Model can read the platform UI text as if the creator said "follow." Manifests as: `cta_present=true` on a 30s vlog where the creator never says anything about following. Drives D-06 penalty mis-application. *Mitigation cue:* dataset must include videos with platform UI visible but no creator CTA, with `cta_present=false` labels.
6. **Niche-blindness — Beauty rubric applied to Comedy.** If the prompt doesn't carry the niche, the model defaults to a "polished, aesthetic, well-lit" lens, which under-scores comedy creators using deliberately scrappy aesthetics and over-scores Lifestyle creators just for being cinematic. Manifests as: Comedy creator gets a low `visual_production_quality` for "messy bedroom shooting" when the messy bedroom is the joke. *Mitigation cue:* prompts inject niche (D-15 already does this), and the eval rubric must be evaluated per-niche, not in aggregate.
7. **Over-penalising videos with collapsed body window (duration ≤8s).** Phase 5 D-08 handles this structurally (skip body segment, mark `gemini_body=false`), but the eval rubric must NOT penalise short videos for "missing body signal" — the absence is correct, not a failure. Manifests as: 8s comedy clip scored low because the aggregator treated `gemini_body=false` as evidence of poor content. *Mitigation cue:* eval rubric explicitly excludes short-video samples from body-segment scoring criteria; reference dataset must include ≥5 short-video examples to surface this case.
8. **Schema-valid but semantically invalid scores.** Gemini returns JSON the schema accepts but a domain expert would reject: `text_overlay_score=8` when there is NO on-screen text (rationale would say "the overlay is impactful" against a non-existent overlay), or `cognitive_load=2` (very low) on a hook with 4 simultaneous text blocks. Manifests as: rationale text contradicts the numeric score. *Mitigation cue:* eval includes a rationale-vs-score consistency check (LLM-judge prompt: "Does the rationale support the score? If rationale says 'no on-screen text' and score > 2, flag.").
9. **Hallucinated multimodal grounding (HAVEN-class failures).** Per [HAVEN benchmark, arXiv 2503.19622](https://arxiv.org/abs/2503.19622), large multimodal models hallucinate at predictable rates against video understanding tasks — object existence, action recognition, and temporal grounding all fail. Phase 5's hook segment is exactly the temporal-grounding case (analyse only 0-5s). Manifests as: model describes content that happens at 12s in the rationale for a hook-only segment, indicating the `videoMetadata` scoping didn't constrain the model's attention as expected. *Mitigation cue:* periodic spot-check that hook-segment rationales reference only 0-5s events; flag any rationale that references later timestamps.

### Regulatory / Compliance Context

This is a **creator-facing coaching tool**. The system analyses creator-uploaded video to produce scores; it does not publish content, does not handle minors' data, does not provide medical/financial/legal advice, and does not display the analysed videos to anyone except the uploading creator. The compliance footprint is narrow but specific:

- **TikTok Community Guidelines (creator-visible only).** The analyser does NOT need to enforce TikTok community guidelines — that's TikTok's job at publish time. BUT the eval rubric should NOT reward content that virally performs *because* it violates guidelines (engagement bait, dangerous challenges, sexually suggestive minors). If the labeled dataset contains a high-engagement video that violates guidelines, it should still be scored against the rubric, but the system's recommendations should not say "do more of this."
- **FTC #ad disclosure (informational only, not enforced by Phase 5).** As of 2025, the FTC requires *in-video* disclosure on TikTok/Reels for any commercial relationship — "ad" / "sponsored" / "#ad" must appear in the video itself, not just the caption. Phase 5 is not a disclosure auditor; this is mentioned only because the `text_overlay_score` rubric should NOT down-score a "#ad" disclosure text as if it were content text. If anything, presence of "#ad" indicates commercial content and the entire prompt context shifts (a sponsored tutorial uses different hook structure than an organic one). *Source:* [FTC Disclosure Requirements 2026 Guide](https://influenceflow.io/resources/influencer-disclosure-requirements-a-complete-2026-guide/), [JoinBrands FTC Guidelines for Influencers](https://joinbrands.com/blog/ftc-guidelines-for-influencers/).
- **Platform watermark policy (cross-platform reach impact).** Per [Stack Influence Instagram 2025 algorithm](https://stackinfluence.com/instagram-2025-algorithm-shake-up/), Instagram down-ranks TikTok-watermarked content by 30-50%. The `visual_production_quality` rubric must NOT score a TikTok watermark as "lower production quality" on a TikTok upload (where it's fine), and the milestone-level watermark detection (ALGO-06, deferred to Phase 9) is the place this signal lives. Phase 5 should detect watermark presence in passing but not weight it negatively in the hook/body rubrics.
- **COPPA — out of scope.** Virtuna is a paid creator tool; users are creators, not viewers. The analyser does not handle viewer data or videos targeting under-13s. If Virtuna later targets under-13 *creators*, COPPA enters scope — currently it does not.
- **GDPR / video retention (Phase 5 inherits, does not add).** Milestone REQUIREMENTS.md INT-05 / INT-06 / INT-07 set the 30-day auto-delete + export policy. Phase 5's segmented analyser does not change the retention surface — it consumes the same uploaded video and discards it after the three Gemini calls settle (Files API delete in outer finally per D-10).

**Net:** no regulatory blocker for shipping Phase 5. Three rubric implications (a) don't reward guideline-violation content, (b) don't down-score #ad disclosures, (c) don't penalise TikTok watermarks on TikTok uploads.

## 2. Framework Decision

**Selected Framework:** Direct `@google/genai` SDK (no agent framework) + Zod at LLM boundary + in-house Promise.allSettled orchestration

**Version:** `@google/genai` v1.41.0 (already in stack) + `zod` v4.3.6 (already in stack)

**Rationale:**
Phase 5 is three parallel scoped `generateContent` calls on a single Files API upload — structured extraction, not an agent. The existing `analyzeVideoWithGemini` at `src/lib/engine/gemini.ts:409` already proves the Files API + `videoMetadata` + `responseSchema` + Zod-at-boundary pattern works in the codebase, so Phase 5 splits one call into three parallel calls of the same shape. Adding any agent framework (LangChain, LangGraph, Google ADK) would:
- Add an abstraction layer that obscures per-segment `AbortController`, per-stage Sentry tagging, per-segment `calculateCost`, and `Promise.allSettled` partial-failure handling — all explicitly required by the milestone.
- Violate the additive-only milestone constraint (no rewrite of `pipeline.ts` or `aggregator.ts`).
- Trip Anti-Pattern #1 from the framework matrix ("Using LangChain for simple chatbots — Direct SDK call is less code, faster, and easier to debug") — which applies double here since the system isn't even a chatbot.

The "no framework" path matches every dimension of this phase: TypeScript hard constraint, model-provider committed (Gemini), system type is extraction not orchestration, and the existing in-house Wave pipeline (Phase 3 D-01) already owns orchestration.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangChain (TS) | Adds heavy abstraction for what is 3 parallel `generateContent` calls; obscures per-segment AbortController/Sentry/cost accounting; matrix Anti-Pattern #1. |
| LangGraph (TS) | Graph-based stateful workflow framework — overkill for 3 stateless parallel extraction calls. No multi-turn state, no branching, no checkpointing requirement. |
| LlamaIndex (TS) | RAG framework — wrong fit. Phase 5 has no retrieval; it's pure multi-modal extraction from an uploaded video file. |
| Google ADK | Python + Java only as of 2026-05-18 — fails the TypeScript hard constraint. Would also lock the analyzer into an agent abstraction when it's a pipeline stage. Reserved for hypothetical future agent-promotion. |
| OpenAI Agents SDK / Claude Agent SDK | Wrong model provider — Gemini multi-modal video upload + `videoMetadata` is Gemini-native. |

**Vendor Lock-In Accepted:** Partial — already locked to Google Gemini for multi-modal video understanding (Files API + `videoMetadata` per-segment scoping is Gemini-native; no cross-provider parity for this capability as of 2026-05-18). Per-segment model overrides (`GEMINI_HOOK_MODEL`, `GEMINI_BODY_MODEL`, `GEMINI_CTA_MODEL` env vars) preserve switching flexibility within the Gemini family (3 Pro ↔ 3.1 Pro when GA ↔ 3 Flash ↔ 3.1 Flash-Lite) without code changes. Aggregator/pipeline layers remain provider-neutral; only the segment modules know about Gemini.

### Entry Point Pattern

The Phase 5 entry point extends the existing `analyzeVideoWithGemini` shape. **Upload once, fan out three parallel `generateContent` calls each scoped to a different `videoMetadata` window, then delete the file after all three settle.**

```typescript
// src/lib/engine/gemini/segmented.ts (new module — orchestrator only)
import { GoogleGenAI, FileState, ApiError } from "@google/genai";
import * as Sentry from "@sentry/nextjs";
import { runHookSegment } from "./hook-segment";
import { runBodySegment } from "./body-segment";
import { runCtaSegment } from "./cta-segment";
import type { GeminiVideoAnalysis, SegmentedAnalysisOptions } from "../types";

const VIDEO_MAX_SIZE_BYTES = 50 * 1024 * 1024;
const VIDEO_POLL_INTERVAL_MS = 500;
const VIDEO_POLL_TIMEOUT_MS = 60_000;

export async function analyzeVideoSegmented(
  videoBuffer: Buffer,
  mimeType: string,
  opts: SegmentedAnalysisOptions,
): Promise<{
  analysis: GeminiVideoAnalysis | null;
  cost_cents: number;
  signalAvailability: { gemini_hook: boolean; gemini_body: boolean; gemini_cta: boolean };
}> {
  if (videoBuffer.byteLength > VIDEO_MAX_SIZE_BYTES) {
    throw new Error("Video exceeds 50MB cap. Trim before uploading.");
  }

  const ai = getClient(); // reuse gemini.ts:81 singleton
  let uploadedFileName: string | undefined;

  try {
    // 1. Single Files API upload (reuse the lines 425-455 pattern verbatim).
    const blob = new Blob([new Uint8Array(videoBuffer)], { type: mimeType });
    const uploaded = await ai.files.upload({ file: blob, config: { mimeType } });
    if (!uploaded.name) throw new Error("Upload returned no file name.");
    uploadedFileName = uploaded.name;

    // 2. Poll for state === ACTIVE (PROCESSING → ACTIVE | FAILED).
    const fileUri = await waitForActive(ai, uploadedFileName);

    // 3. Fan-out: three parallel scoped generateContent calls on the SAME fileUri.
    //    Per-segment AbortController so a slow segment doesn't block siblings.
    //    Each segment owns its own model, prompt, schema, Zod validation, cost.
    const [hookSettled, bodySettled, ctaSettled] = await Promise.allSettled([
      runHookSegment(ai, fileUri, mimeType, opts),
      runBodySegment(ai, fileUri, mimeType, opts),
      runCtaSegment(ai, fileUri, mimeType, opts),
    ]);

    return mergeSegments(hookSettled, bodySettled, ctaSettled, opts);
  } finally {
    // 4. Delete the file AFTER all three settle (success or fail). Best-effort.
    if (uploadedFileName) {
      try {
        await ai.files.delete({ name: uploadedFileName });
      } catch {
        // Swallow — file expires server-side anyway.
      }
    }
  }
}

async function waitForActive(ai: GoogleGenAI, name: string): Promise<string> {
  const start = Date.now();
  // eslint-disable-next-line no-constant-condition
  while (true) {
    if (Date.now() - start > VIDEO_POLL_TIMEOUT_MS) {
      throw new Error("Files API processing timed out after 60s.");
    }
    const info = await ai.files.get({ name });
    if (info.state === FileState.ACTIVE && info.uri) return info.uri;
    if (info.state === FileState.FAILED) {
      throw new Error("Files API marked the upload as FAILED.");
    }
    await new Promise((r) => setTimeout(r, VIDEO_POLL_INTERVAL_MS));
  }
}
```

const HOOK_MODEL = process.env.GEMINI_HOOK_MODEL ?? "gemini-3-pro";
const HOOK_TIMEOUT_MS = 30_000;

export async function runHookSegment(
  ai: GoogleGenAI,
  fileUri: string,
  mimeType: string,
  opts: SegmentedAnalysisOptions,
) {
  const controller = new AbortController();
  const timeout = setTimeout(() => controller.abort(), HOOK_TIMEOUT_MS);
  try {
    const response = await ai.models.generateContent({
      model: HOOK_MODEL,

          role: "user",

              videoMetadata: { startOffset: "0s", endOffset: "5s" },
            },

        },
      ],

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `ai.files.upload(...)` → poll `ai.files.get` until `state === FileState.ACTIVE` | Two-step upload + processing handshake. Video files start in `PROCESSING`; they cannot be referenced until `ACTIVE`. | Once per `analyzeVideoSegmented` call, before any `generateContent` fans out. Pattern at `gemini.ts:425-455` is the canonical reference. |
| `videoMetadata: { startOffset, endOffset }` inside `parts[].fileData` sibling | Per-call segment scoping. Tells Gemini to sample only the byte range corresponding to `startOffset..endOffset`. Format is `"<seconds>s"` as a string (e.g. `"5s"`, `"22s"`). camelCase keys in JS/TS. Default `fps` is 1. | Inside each segment helper's `contents[].parts[]` array. Changing it per call is what lets one upload power 3 different scopes. |
| `responseMimeType: "application/json"` + `responseSchema` (or `responseJsonSchema`) | Structured-output mode. Gemini constrains its decode to match the schema, eliminating most "extra prose around JSON" failures. Schema dialect is a strict subset of OpenAPI 3.0: `STRING / NUMBER / INTEGER / BOOLEAN / ARRAY / OBJECT / NULL`, plus `enum`, `required`, `nullable`, `propertyOrdering`. **No discriminated unions, no `oneOf`, no recursive `$ref`.** | Every Phase 5 segment call. The CTA presence-aware shape is expressed by making `strength` and `type` `nullable: true` rather than via a union of two object shapes. |
| `AbortController` + `config.abortSignal` | Per-call cooperative cancellation. **Client-only** — Google still bills any tokens already generated when you abort. | Wrap every segment call so a single slow shard cannot stall the other two. Use a fresh controller per segment, not one shared across all three. |
| `response.usageMetadata.{promptTokenCount, candidatesTokenCount}` | Server-reported token usage. Video frames count as input tokens (sampled at 1 fps by default — a 5s hook segment ≈ 5 × ~258 = ~1290 input tokens of video plus prompt text). | Inside per-segment `calculateCost(model, usageMetadata)`. The new helper signature must take `model` because Gemini 3 Pro and Gemini 3 Flash have different price tables (see Section 4). |
| `ApiError` (thrown by SDK) | Single error class from `@google/genai` carrying `name`, `message`, `status`. Use `e instanceof ApiError && e.status === 429` to branch on rate-limit retryability. | Inside each segment's catch block, before falling back to a null return + Sentry capture + `pipeline_warning` per D-08. |

### Common Pitfalls

1. **Deleting the Files API upload before all three calls finish.** The existing `gemini.ts:532-541` `finally` block deletes the file after one call returns. Phase 5 has THREE concurrent calls on the same `fileUri`. If `finally` lives inside the orchestrator (around `Promise.allSettled`), it fires after all three settle and the pattern is safe. **If `finally` lives inside any per-segment helper, the first helper to settle will delete the file while the other two are mid-request, causing immediate 404 / PERMISSION_DENIED on the survivors.** The delete *must* sit in the outer orchestrator's `finally`, never inside the segment helpers.
2. **Firing `generateContent` before the file is `ACTIVE`.** A naive `Promise.all([upload, hookCall, bodyCall, ctaCall])` fan-out without an explicit poll loop will race the file-processing step and yield "FILE_STATE_UNSPECIFIED" or 400 errors. Always `await` the `state === FileState.ACTIVE` poll loop *before* the three-way fan-out starts. The existing while-loop at `gemini.ts:444-455` is correct; reuse it verbatim and only move it into `analyzeVideoSegmented`.
3. **Misplacing `videoMetadata`.** It is a sibling of `fileData` *inside the same `Part` object*, not a top-level config or a sibling of `parts`. Correct: `{ parts: [{ text }, { fileData: {...}, videoMetadata: { startOffset: "0s", endOffset: "5s" } }] }`. Wrong: `{ parts: [...], videoMetadata: {...} }` or `{ parts: [{ text }, { fileData }, { videoMetadata }] }`. The Part type tolerates a videoMetadata-only part at the type level but the API ignores it unless it is on the same Part as the fileData/inlineData.
4. **Format errors on `startOffset` / `endOffset`.** Strings only, with the trailing `"s"` unit literal (e.g. `"5s"`, `"22s"`, `"57s"`). Numbers (`5`), unitless strings (`"5"`), ISO 8601 durations (`"PT5S"`), and `HH:MM:SS` timecodes (`"00:00:05"`) all silently disable the segment scoping in some client versions and the model will analyze the full video. Always assert the helper builds them via a template literal like `${seconds}s` and unit-test with a value > 9 to catch zero-padding bugs.
5. **One shared `AbortController` for the whole `Promise.allSettled`.** If one segment times out and aborts the shared controller, the other two are killed mid-flight even though they still had budget. Make three controllers, one per segment, each with its own `setTimeout`. The lib design here is the same as the existing `gemini.ts:466-469` pattern — repeat it three times, do not factor it out into a single controller.
6. **Forgetting that `abortSignal` does not stop server-side billing.** The SDK doc states explicitly: *"AbortSignal is a client-only operation. Using it to cancel an operation will not cancel the request in the service. You will still be charged usage for any applicable operations."* Treat aborts as cost-bearing failures in the per-segment `cost_cents` accounting — do not assume a timed-out call costs zero.
7. **Discriminated unions / `oneOf` in the CTA schema.** Gemini's `responseSchema` dialect does *not* support `oneOf`. The presence-aware CTA shape (`{cta_present: true, strength: 7.2, type: "follow"}` vs `{cta_present: false, strength: null, type: null}`) must be expressed as a single OBJECT with `nullable: true` on `strength` and `type` and a boolean `cta_present` discriminator. Use `propertyOrdering: ["cta_present", "strength", "type", "rationale"]` so the model emits `cta_present` first, which empirically improves the null-when-absent rate.
8. **Asserting on `response.text` after using `responseSchema`.** Although the Gemini docs claim `response.text` is "guaranteed valid JSON," in practice the field can still be undefined when the model refuses (safety filters, max_output_tokens too low, content-policy block). Always default with `response.text ?? ""` and let the Zod parse step produce the actionable error, exactly as `parseGeminiVideoResponse` does at `gemini.ts:108-115`.
9. **Reusing the calibration-baseline cache singleton across worker processes.** `loadCalibrationData()` at `gemini.ts:120-139` caches in a module-level `cachedCalibration` variable. Multiple Vercel function instances each hold their own cache — fine, but the first request in each cold start pays the disk read. Phase 5 does not change this; just be aware that adding the segmented entry point still inherits the same warm-cache assumption.

export async function runBodySegment(

  mimeType: string,
  opts: SegmentedAnalysisOptions,

  const t0 = performance.now();
  const controller = new AbortController();
  const timeout = setTimeout(() => controller.abort(), SEGMENT_TIMEOUT_MS);
  const model = process.env.GEMINI_BODY_MODEL ?? "gemini-3-flash";

  try {
    const response = await ai.models.generateContent({
      model,

        {
          role: "user",

            { text: buildBodyPrompt(opts.calibration, opts.niche, opts.contentType) },

              videoMetadata: {
                startOffset: `${opts.bodyStartSec}s`,   // 5s
                endOffset: `${opts.bodyEndSec}s`,       // duration - 3s
              },
            },
          ],
        },
      ],
      config: {
        responseMimeType: "application/json",
        responseSchema: BODY_SEGMENT_GEMINI_SCHEMA,
        abortSignal: controller.signal,
        maxOutputTokens: MAX_OUTPUT_TOKENS,
      },
    });

    // Zod-at-boundary — never trust raw response.text. (Same pattern as gemini.ts:108-115.)
    const analysis = BodySegmentZodSchema.parse(
      JSON.parse(stripFences(response.text ?? "")),
    );
    const cost_cents = calculateCost(model, response.usageMetadata);
    const duration_ms = Math.round(performance.now() - t0);

    // Per-segment event emission for Phase 3 D-01 SSE contract.
    opts.onStageEvent?.({
      type: "stage_end",
      stage: "gemini_body",
      wave: 1,
      duration_ms,
      cost_cents: +cost_cents.toFixed(4),
      success: true,
    });

    return { ok: true, analysis, cost_cents, model };
  } catch (error) {
    Sentry.captureException(error, {
      tags: { stage: "gemini_body", model },
    });
    opts.onStageEvent?.({
      type: "stage_end",
      stage: "gemini_body",
      wave: 1,
      duration_ms: Math.round(performance.now() - t0),
      cost_cents: 0,
      success: false,
      error: error instanceof Error ? error.message : String(error),
    });
    return { ok: false, error };          // null result handled in mergeSegments

The merge step is plain TypeScript — no SDK calls, no I/O:

```typescript
function mergeSegments(
  hook: PromiseSettledResult<SegmentResult<HookSegmentResult>>,
  body: PromiseSettledResult<SegmentResult<BodySegmentResult>>,
  cta:  PromiseSettledResult<SegmentResult<CtaSegmentResult>>,
  opts: SegmentedAnalysisOptions,
) {
  const hookOk = hook.status === "fulfilled" && hook.value.ok;
  const bodyOk = body.status === "fulfilled" && body.value.ok;
  const ctaOk  = cta.status  === "fulfilled" && cta.value.ok;

  // D-09: all three failed → null analysis + warning.
  if (!hookOk && !bodyOk && !ctaOk) {
    opts.onStageEvent?.({
      type: "pipeline_warning",
      message: "All Gemini segments failed — video signal unavailable.",
      stage: "gemini_video_unavailable",
    });
    return {
      analysis: null,
      cost_cents: 0,
      signalAvailability: { gemini_hook: false, gemini_body: false, gemini_cta: false },
    };
  }

  // D-08: partial-failure warnings (one per failed segment).
  for (const [name, ok] of [["hook", hookOk], ["body", bodyOk], ["cta", ctaOk]] as const) {
    if (!ok) opts.onStageEvent?.({
      type: "pipeline_warning",
      message: `Gemini ${name} analysis unavailable — score uses other segments.`,
      stage: `gemini_${name}`,
    });
  }

  // Hook segment owns the 5 TikTok factors + hook_decomposition.
  // Body fills 3 of 4 video_signals (hook_visual_impact passthrough handled below).
  // CTA fills cta_segment.
  return {
    analysis: {
      factors: hookOk ? hook.value.analysis.factors : DEFAULT_NULL_FACTORS, // see schemas.ts
      overall_impression: hookOk ? hook.value.analysis.overall_impression : "",
      content_summary: hookOk ? hook.value.analysis.content_summary : "",
      video_signals: {
        visual_production_quality: bodyOk ? body.value.analysis.video_signals.visual_production_quality : null,
        hook_visual_impact:        hookOk ? hook.value.analysis.hook_decomposition.visual_stop_power : null,
        pacing_score:              bodyOk ? body.value.analysis.video_signals.pacing_score : null,
        transition_quality:        bodyOk ? body.value.analysis.video_signals.transition_quality : null,
      },
      hook_decomposition: hookOk ? hook.value.analysis.hook_decomposition : null,
      cta_segment:        ctaOk  ? cta.value.analysis  : null,
    },
    cost_cents:
      (hookOk ? hook.value.cost_cents : 0) +
      (bodyOk ? body.value.cost_cents : 0) +
      (ctaOk  ? cta.value.cost_cents  : 0),
    signalAvailability: { gemini_hook: hookOk, gemini_body: bodyOk, gemini_cta: ctaOk },
  };
}
```

**Tool Use:**

No tool/function-calling required — Phase 5 uses pure structured-output extraction (`responseMimeType: "application/json"` + `responseSchema`). The "tools" Phase 5 *does* configure on each call are:

- `responseSchema` — the OpenAPI-3-subset schema literal (`HOOK_SEGMENT_GEMINI_SCHEMA`, `BODY_SEGMENT_GEMINI_SCHEMA`, `CTA_SEGMENT_GEMINI_SCHEMA`).
- `abortSignal` — per-segment controller.
- `maxOutputTokens` — hard 1500 ceiling. Prevents the Pro hook call from drifting into multi-thousand-token rationales that blow the 1.3¢ budget.
- `safetySettings` — leave SDK default. The corpus is TikTok content; flipping safety thresholds risks dropping legitimate hooks (e.g. comedy with mild profanity).

No `tools: [...]` array, no function declarations, no MCP — every signal Phase 5 emits comes from the model's structured-output JSON, not from agent tool calls.

**State Management:**

Stateless within the pipeline stage — each `analyzeVideoSegmented(buffer, mimeType, opts)` call is self-contained and returns its merged result for the orchestrator to write back into the existing `analysis_results` row. Persistent state lives where it already does:

- **Per-call** — `AbortController`, `Promise.allSettled` results, merged `GeminiVideoAnalysis`. Lives in the function's stack frame, gone when the function returns.
- **Cross-call (read-only)** — `cachedCalibration` module singleton at `gemini.ts:79` (preserved). Phase 5 reuses it for the body and hook prompts. Three segments do NOT each pay the disk-read cost — they share the warm cache.
- **Cross-call (writes)** — `analysis_results.signal_availability` JSONB persists the three new keys (`gemini_hook`, `gemini_body`, `gemini_cta`) per D-12. Existing `gemini` key derived as `gemini_hook || gemini_body || gemini_cta`. Persistence happens at the pipeline level (already wired), Phase 5 only widens the shape.
- **Inter-segment** — none. The three segment helpers do NOT share state, do NOT communicate, do NOT depend on each other's outputs. This is what enables `Promise.allSettled` without coordination overhead.

**Context Window Strategy:**

Gemini 3 Pro and Flash both expose a 1M-token context window. Phase 5 never approaches it — three reasons:

1. **`videoMetadata` slices the input** — the hook call's input is ~1.3K video tokens + ~500 text tokens of prompt + rubric; the body call is ~5.7K + ~400; the CTA call is ~0.8K + ~300. Total across all three: under 10K input tokens. Worst-case 3-minute video (50MB cap) on the body call is ~177s × 258 tokens/s = ~45K — still 2% of the 1M budget.
2. **No conversation history** — each segment is a single-turn `generateContent`. There is no chat memory, no `system_instruction` accumulation, no prior-message replay.
3. **Output is bounded** — `maxOutputTokens: 1500` keeps the rationale prose from blowing the budget even on the Pro segment.

The interesting context-window question is **prompt-prefix reuse for caching**: the hook segment's system prompt (rubric + calibration baseline + 6-score schema description) is byte-identical across all users. Gemini's native context-caching mechanism (Files-API-backed cached prompts) could turn that ~500-token prefix into a free cache hit on every subsequent call. **Deferred to a future phase** (CONTEXT.md `<deferred>`) — current per-call cost (~1.3¢) is already inside budget and the cache complexity is not justified pre-Phase 10. Section 4b.4 below documents the deferred opportunity and the trigger condition for revisiting.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### Structured Outputs with Zod (TypeScript equivalent of Pydantic)

**This codebase uses Zod at the LLM boundary** — `Pydantic` is the Python idiom, `Zod` is its TypeScript counterpart. The existing `parseGeminiVideoResponse` helper at `gemini.ts:108-115` already implements the pattern; Phase 5 extends it from one global schema to three per-segment schemas.

The discipline is two-layer:

1. **`responseSchema` (OpenAPI-3 subset)** — sent to Gemini so the *decoder* is constrained to the shape. Eliminates ~95% of "extra prose around JSON" failures.
2. **Zod schema** — applied client-side to the model's `response.text`. Catches the ~5% the decoder still slips through (schema-valid JSON with semantically invalid values: score=11.5 instead of 0-10, weakest_modality naming a modality not in the four-option enum, etc.).

**Per-segment Zod models for Phase 5:**

```typescript
// src/lib/engine/gemini/schemas.ts
import { z } from "zod";

// --- Hook segment ---------------------------------------------------------
const ScoreSchema = z.number().min(0).max(10);

export const HookDecompositionZod = z.object({
  visual_stop_power: ScoreSchema,
  audio_hook_quality: ScoreSchema,
  text_overlay_score: ScoreSchema,
  first_words_speech_score: ScoreSchema,
  weakest_modality: z.enum([
    "visual_stop_power",
    "audio_hook_quality",
    "text_overlay_score",
    "first_words_speech_score",
  ]),
  visual_audio_coherence: ScoreSchema,
  cognitive_load: ScoreSchema,
});

export const FactorZod = z.object({
  name: z.enum([
    "Scroll-Stop Power",
    "Completion Pull",
    "Rewatch Potential",
    "Share Trigger",
    "Emotional Charge",
  ]),
  score: ScoreSchema,
  rationale: z.string().min(1).max(500),
  improvement_tip: z.string().min(1).max(500),
});

export const HookSegmentZodSchema = z.object({
  factors: z.array(FactorZod).length(5),
  overall_impression: z.string().min(1).max(800),
  content_summary: z.string().min(1).max(800),
  hook_decomposition: HookDecompositionZod,
});

// --- Body segment ---------------------------------------------------------
export const BodySegmentZodSchema = z.object({
  video_signals: z.object({
    visual_production_quality: ScoreSchema,
    pacing_score: ScoreSchema,
    transition_quality: ScoreSchema,
  }),
  body_summary: z.string().min(1).max(800),
});

// --- CTA segment (presence-aware, expressed via nullable, NOT discriminated union) ---
export const CtaSegmentZodSchema = z
  .object({
    cta_present: z.boolean(),
    strength: ScoreSchema.nullable(),
    type: z
      .enum(["follow", "comment", "link_in_bio", "watch_next", "engage_question", "other"])
      .nullable(),
    rationale: z.string().min(1).max(400),
  })
  // Cross-field invariant — Gemini's responseSchema can't express this; Zod can.
  .refine(
    (v) => (v.cta_present ? v.strength !== null && v.type !== null : v.strength === null && v.type === null),
    { message: "When cta_present=true, strength and type must be non-null; when false, both must be null." },
  );

export type HookSegmentResult = z.infer<typeof HookSegmentZodSchema>;
export type BodySegmentResult = z.infer<typeof BodySegmentZodSchema>;
export type CtaSegmentResult  = z.infer<typeof CtaSegmentZodSchema>;
```

**How Gemini integrates** — Zod is *not* fed directly to the SDK; the SDK takes a literal OpenAPI-3-subset object via `responseSchema` (or `responseJsonSchema`). The pattern is to hand-write the Gemini schema once per segment alongside the Zod schema, then *unit-test that the two are in sync* (sample valid Zod outputs and assert they would validate against the Gemini schema, and vice versa). This is one of the few places where DRY-violation is correct — the two schemas serve different layers (decoder constraint vs runtime validation) and chasing a single source of truth via `zod-to-json-schema` introduces a runtime dependency on a converter library that has historically over-emitted unsupported keywords (`$schema`, `additionalProperties`, `$ref`) that Gemini rejects with a 400 INVALID_ARGUMENT.

**Retry logic:**
- **0 schema-validation retries on the Pro hook segment.** A schema failure on Pro is loud evidence of a structural prompt bug; the per-segment cost is too high to silently retry. Capture to Sentry with `tags: { stage: "gemini_hook", failure_class: "zod_validation" }` and return null (D-08 graceful degradation handles the user-facing impact).
- **1 schema-validation retry on the Flash body/CTA segments** with a corrective preamble (`"Your previous response was not valid JSON matching the schema. Return ONLY the JSON object."`). This matches the existing `MAX_RETRIES = 2` text-call pattern at `gemini.ts:18` but with a tighter budget for video calls. Log every retry attempt at `level: "warn"` with the original validation error message — repeat failures indicate the prompt drift problem, not transient model noise.
- **3 total attempts (1 retry × 2 backoff) ONLY for transient `ApiError` with `status` in `[429, 500, 502, 503, 504]`.** Exponential backoff: 1s, 3s (mirrors `gemini.ts:389`). All other `ApiError` statuses (400 INVALID_ARGUMENT, 403 PERMISSION_DENIED, 404 NOT_FOUND, 413 REQUEST_ENTITY_TOO_LARGE) are non-retryable — fail fast, capture to Sentry, return null.
- **Never retry on `AbortError`** (timeout). The user's request budget is already blown; absorb the partial result.

### Async-First Design

The codebase is end-to-end async — every Gemini call is `await`-ed inside an `async function`, and the orchestrator returns a Promise. Phase 5's discipline:

- **`Promise.allSettled` over `Promise.all`** — D-08 graceful-degradation requires per-segment failure isolation. `Promise.all` rejects the whole tuple on the first error and short-circuits the other two in-flight calls (they continue running on the server and are billed but their results are discarded). `Promise.allSettled` returns all three results regardless of individual failure, which is the explicit Phase 5 contract.
- **Fan-out is the parallelism primitive** — three `runXSegment(...)` calls inside the `Promise.allSettled([])` argument array. Do NOT chain them with `await runHook(); await runBody(); await runCta();` — that serializes the latency to ~3× single-segment time, blows the 30s wave-1 budget, and defeats the architectural point of the segmentation refactor.
- **`AbortController` is per-segment, not shared** — three controllers, three `setTimeout`s, three `clearTimeout`s in three `finally` blocks. One global controller would cascade a single timeout into three cancellations.
- **Stream vs await** — Phase 5 uses `await` exclusively. Streaming (`ai.models.generateContentStream`) is the right tool when the user is reading the LLM's output token-by-token in a chat UI. It is the *wrong* tool for structured-output extraction: you cannot Zod-validate a partial JSON document, and the merge step needs all three segments to be complete objects before it can fill the `GeminiVideoAnalysis` shape. The /api/analyze SSE *does* stream per-segment events to the client via `onStageEvent`, but the events are typed orchestration events (`stage_start` / `stage_end` / `pipeline_warning`), not LLM token deltas.
- **The one common mistake** — wrapping the three segment calls in a `for` loop with `await` inside the loop body. This is the JavaScript equivalent of Python's `asyncio.run()` inside an event loop: it *looks* concurrent because the helper functions are async, but `for...of + await` is strictly sequential. The Vitest test for `analyzeVideoSegmented` must assert wall-clock latency ≤ max(hook, body, cta) + 200ms overhead — a serialized implementation would be ~3× that and the test catches it.

```typescript
// Right: parallel fan-out.
const [hook, body, cta] = await Promise.allSettled([
  runHookSegment(ai, fileUri, mimeType, opts),
  runBodySegment(ai, fileUri, mimeType, opts),
  runCtaSegment(ai, fileUri, mimeType, opts),
]);

// Wrong: looks async, runs sequentially.
const results = [];
for (const fn of [runHookSegment, runBodySegment, runCtaSegment]) {
  results.push(await fn(ai, fileUri, mimeType, opts));   // ← await inside loop = serial
}
```

### Prompt Engineering Discipline

**System vs user prompt separation:**

Gemini's `generateContent` does not have a dedicated `system` role for the Gemini-1/2/3 Developer API the way Anthropic and OpenAI do — there is a `systemInstruction` field on `generateContent` config in newer SDK versions, but `gemini.ts:154-198` builds the whole prompt as one string with a `---` separator (`${systemPrompt}\n\n---\n\n${userMessage}`). **Phase 5 keeps the same separator convention but tightens the discipline:**

- **System prompt (stable, byte-identical across calls)**: the rubric. For the hook segment, this is the 6-score decomposition definitions + "Score 0-10 with one decimal" rule + the 5 TikTok factors definitions + the calibration baseline (viral threshold, p90 weighted engagement, duration sweet spot, top 3 differentiators). This block is the same for *every* hook call ever made — it is the cacheable prefix.
- **User prompt (per-call context)**: niche, content_type, creator context (Card 4 style + Card 5 references when budget allows), and **the segment scope statement**. The phrase *"Analyze ONLY the 0-5s hook of this video"* must live in the user prompt, not in the system prompt, because (a) the `videoMetadata` already constrains the input to that window — the prompt is reinforcement — and (b) the cacheable prefix must not contain per-call window text or the cache hit rate drops to zero.

```typescript
// hook-segment.ts buildHookPrompt — schematic
function buildHookPrompt(opts: SegmentedAnalysisOptions): string {
  const systemPrompt = buildHookSystemPrompt(opts.calibration); // <-- cacheable, ~500 tokens
  const userPrompt = `
Niche: ${opts.niche}
Content type: ${opts.contentType}
${opts.creatorContext?.card4_content_style ? `Creator style: ${opts.creatorContext.card4_content_style}` : ""}

Analyze ONLY the 0-5s hook of this video. Score each of the 5 factors AND each of the 4 sub-modalities (visual_stop_power, audio_hook_quality, text_overlay_score, first_words_speech_score). Then name the WEAKEST of the four sub-modalities. Then score visual_audio_coherence and cognitive_load on the same 0-10 scale.

If a modality is genuinely absent from the hook (e.g. no on-screen text, no spoken words in the first 5s), score it at 0 and mention this in the rationale. Do NOT hallucinate scores for modalities the video doesn't contain.
`.trim();
  return `${systemPrompt}\n\n---\n\n${userPrompt}`;
}
```

**Few-shot guidance** — Phase 5 does **not** use few-shot examples in the prompts. Three reasons:

1. The rubric itself (with calibration baselines) gives the model enough scoring anchors.
2. Few-shot at the Pro tier would inflate the input-token cost on the most expensive segment by ~30%, breaking the per-call budget.
3. Few-shot examples encode the labeler's bias — for the Phase 12 eval-harness benchmark to be honest, the model must score from rubric alone, not from "examples that look like the test set."

If the eval-harness benchmark surfaces consistent under/over-scoring on a specific sub-score, the corrective is to refine the rubric language in the system prompt (which is auditable), not to add few-shot demonstrations (which embed bias invisibly).

**Token budget strategy:**

- `maxOutputTokens: 1500` on every segment — hard ceiling.
- Per-segment soft warning at `cost_cents > {1.6 hook | 0.7 body | 0.4 cta}`. Triggers a `log.warn` with the full token breakdown so prompt drift is visible in production logs.
- Per-video hard cap at 5.0¢ in `mergeSegments` — if the sum exceeds it, emit a `pipeline_warning` (does NOT fail the result; the warning is for ops monitoring per CONTEXT.md `<deferred>` "Per-segment Sentry breadcrumbs ↔ Phase 10 ML audit").

### Context Window Management

This is a multi-modal extraction system on bounded video inputs (50MB / ~3min cap), not RAG, not conversational, not autonomous. The context-window dimensions Phase 5 cares about are:

- **Video token cost per second.** Gemini samples video at 1 fps by default → ~258 tokens per video-second. 30s video = ~7740 video tokens; 3-min video = ~46K. Both fit comfortably in the 1M context window of Gemini 3 Pro/Flash. **Phase 5 does NOT need to truncate or chunk video.** The `videoMetadata` segmentation IS the chunking strategy.
- **`videoMetadata` is per-call truncation.** Each segment call's input is bounded by `endOffset - startOffset` seconds of video, regardless of the underlying file's total length. This is more efficient than re-uploading three trimmed clips because:
  - One upload pays the file-processing latency once (~2-10s).
  - One upload counts against the rate limit once.
  - Storage and bandwidth cost is paid once.
- **Prompt prefix reuse (DEFERRED).** Each segment's system prompt (rubric + calibration baseline) is byte-identical across all users. Gemini's native context-caching mechanism (`ai.caches.create({ contents: [...], model, ttl: ... })` → reference via `cachedContent` field in `generateContent`) could turn the ~500-token hook prompt prefix into a cache hit and cut input-token cost on the heaviest segment by ~30%. **Deferred** — current ~1.3¢ per hook call is inside budget; the cache complexity (TTL management, cache-name lookup, cold-start path, schema-version invalidation on rubric updates) is not justified pre-Phase 10. Trigger to revisit: hook-segment cost trends above 1.5¢/call in production telemetry (Phase 10 ML audit).
- **No multi-turn conversation, no chat history, no summarisation needed.** Each segment is one-shot.
- **Short-video edge case.** When video duration < 8s (5s hook + 3s CTA), the body window collapses or inverts. Planner's discretion picks (CONTEXT.md `<decisions>` D-08 recommendation = option a): skip the body call, mark `gemini_body=false`, ship hook + CTA only. This is the natural extension of the partial-failure pattern and avoids special-casing in the schema.

Total milestone cost budget is $0.075/video — Phase 5 consumes 2.0¢ of it, leaving 5.5¢ headroom for Phase 4 Wave 0 (~0.4¢), DeepSeek strategic analysis (~1¢), and future stages (Phase 6 audio, Phase 8 retrieval, Phase 10 aggregator extension).

**Per-video latency budget (target ≤ 22s p95, Wave 1 SLO):**

- Files API upload + poll-to-ACTIVE: 2-10s (network-bound; dominated by file size).
- Three parallel `generateContent` calls: max(hook ~15s, body ~12s, cta ~5s) = ~15s p95. Parallel fan-out is what holds total wave latency at the max-of-three, not the sum.
- Files API delete: best-effort, fire-and-forget within `finally`. Not on the critical path.
- **Total ~17-25s p95.** Each segment's `AbortController` enforces a 30s ceiling. The merge step is synchronous, sub-millisecond.

**Caching strategy:**

- **Exact-match cache: applies to the eval-harness only.** Production analysis is per-user, per-video — no two production calls share a video buffer. The eval-harness corpus-replay path *does* benefit from exact-match caching (same video file, same prompt → same Gemini response), but this is handled at the eval harness level (Phase 1's `bypassCache` semantics), not inside `analyzeVideoSegmented`.
- **Semantic cache: not applicable.** No retrieval, no query-vector reuse, no near-duplicate video heuristic worth the implementation cost at current corpus scale.
- **Gemini native context cache for prompt prefix: DEFERRED** (see Section 4b.4 above). Trigger condition: hook-segment cost trends above 1.5¢/call in Phase 10 telemetry.

**Sub-task model routing (already encoded in D-01):**

- Reasoning-heavy task (hook decomposition: 6 scores + weakest_modality + coherence + cognitive_load) → Pro tier.
- Description / 0-10 scoring task (body video_signals) → Flash tier (3-6× cheaper, equivalent quality on bounded tasks).
- Presence-detection + classification (CTA segment) → Flash tier.
- Env-var routing (`GEMINI_HOOK_MODEL` / `GEMINI_BODY_MODEL` / `GEMINI_CTA_MODEL`) gives ops the single-flip upgrade path when Gemini 3.1 Pro / 3.1 Flash-Lite / 3.2 Flash go GA — no code change, no redeploy, just an env-var update + restart.

### Eval Tooling

**Primary Tool:** **Extend the existing in-house eval harness** (`src/lib/engine/corpus/eval-harness.ts` + `src/lib/engine/corpus/eval-runner.ts` + `pnpm eval`). Tracing/observability layer uses the **already-installed Sentry + structured-logger + `emitStageStart`/`emitStageEnd` per-stage events** from Phase 3 (D-01/D-02). **No new tracing platform.**

**Rationale for NOT adopting Arize Phoenix (or Langfuse / RAGAS):**

1. **The project already has tracing.** Phase 3 D-01 ships `emitStageStart`/`emitStageEnd` per stage with `duration_ms`, `cost_cents`, `success`, `error` — Phase 5 emits these three times (hook/body/cta) per call. Sentry already receives per-stage `tags: { stage }` captures (`gemini.ts:526`). Phoenix would re-implement what's already wired via OpenTelemetry shims and double the per-call write cost.
2. **Phoenix's RAG-focused metrics don't apply.** Phase 5 is *structured extraction*, not RAG — there's no retrieval, no context faithfulness, no answer relevance. The eval dimensions are domain-specific Section 1b rubrics, not RAGAS metrics.
3. **The corpus replay harness IS the eval platform.** Phase 1's `eval-runner.ts` already runs predictions across the corpus, computes macro-F1, persists `benchmark_results` rows tagged with `engine_version + corpus_version`. Phase 5 only widens what each row carries (the 6 hook decomposition scores + cta_segment + signalAvailability per-segment flags). This is the established eval surface every prior phase has used; introducing Phoenix in parallel violates the additive-only milestone constraint.
4. **Future revisit trigger:** If/when a non-engine AI surface ships (e.g., agentic creator-coaching multi-turn chat in a later milestone), revisit Phoenix as the tracing layer for *that* surface. Phase 5 stays inside the existing stack.

## 7. Production Monitoring

**Tracing Tool:** **Sentry + structured logger + per-stage `emitStageStart` / `emitStageEnd` events** — the stack already wired in Phase 3 (D-01/D-02) and previously across the codebase (`gemini.ts:526`, `src/lib/logger.ts`). NOT Arize Phoenix. NOT Langfuse.

**Justification for staying inside the existing observability stack:**

1. **Per-stage event schema already covers what tracing needs.** Phase 3 D-02 ships `StageEvent { type, stage, wave, duration_ms, cost_cents, success, error }`. Phase 5 emits these three times per call (`gemini_hook`, `gemini_body`, `gemini_cta`). The SSE stream surfaces them to /api/analyze; the same payload writes to Sentry breadcrumbs via `Sentry.captureMessage` at the orchestrator level. Phoenix would re-implement this via OpenTelemetry shims with no observability gain.
2. **Sentry already has per-stage tags.** `Sentry.captureException(err, { tags: { stage: "gemini_hook" } })` is the established pattern. Phase 5 inherits it for each of the three new stage names. Sentry's issue grouping by tag gives the same per-segment failure breakdown Phoenix would, with zero new infrastructure cost.
3. **Adding Phoenix would violate the additive-only milestone constraint.** Phoenix's value is RAG + multi-agent tracing — neither applies here (Section 5 "Eval Tooling" rationale).
4. **Future revisit trigger:** introduce Phoenix only when (a) a multi-agent surface ships (e.g., creator-coaching chat in a future milestone) AND (b) Sentry + structured logs prove insufficient for that surface. For Phase 5, the existing stack is provably sufficient — every signal the eval harness + flywheel reads is already emitted.

### Alert Thresholds

Alerts route to the on-call channel (Sentry → existing alert routing). Thresholds tuned to fire on real regressions, not normal variance.

|-------|-----------|----------|
| Per-segment schema-validation fail rate | > 1% per segment over 1h | **PAGE** — likely prompt or SDK regression |
| Per-segment p95 latency | > 90s for any segment over 1h | **PAGE** — Gemini-side slowdown or our retry path looping |
| Per-segment cost above soft cap (hook 1.6¢ / body 0.7¢ / cta 0.4¢) | sustained > 5% of calls over 24h | **WARN** — prompt drift investigation |
| Per-segment cost above hard cap (hook 2.0¢ / body 1.0¢ / cta 0.6¢) | any single occurrence in 1h | **WARN** — per-call breadcrumb; cluster of > 10/h → PAGE |

### Smart Sampling Strategy

Production calls flow to human review under signal-based filters — never random sampling at this scale (30-video labeled set is the reference; sampling beyond it must be cost-aware). All sampled rows are persisted to `benchmark_failure_cases` (Phase 1 deferred-Claude-discretion bucket) with the full per-segment payload.

| Filter | Rate / Cadence | Why |
|--------|----------------|-----|
| **Partial-failure rows** (1-of-3 or 2-of-3 segments failed) | 100% sampled (every row) | Failure-pattern catalog for F4 flywheel and Section 1 #4 audit trail |
| **All-3-success cost outliers** (per-video cost in top 1% of last 24h) | 100% of top-1% | Cost-runaway forensic (Section 1 #5); fuels F6 + G2 review |
| **Niche outliers** (analyses in any niche cell < 5% of 24h volume) | 100% sampled | Catches niche-blindness regressions (Section 1b failure mode #6) in low-volume cells before they propagate |
| **Forced-CTA candidates** (`cta_segment.cta_present=true` AND `wave0Result.content_type.type ∈ {comedy, slideshow}`) | 100% sampled | Section 1 #3 (forced-CTA false positive) — comedy+CTA combo is the highest-value diagnostic row |
| **Brand-watermark suspect rows** (`text_overlay_score ≥ 7` AND content_type ∈ {tutorial, b_roll}) | 50% sampled | Section 1b failure mode #2 watermark-vs-content-text drift |
| **Absent-modality risk** (any hook sub-modality score = 0 AND `weakest_modality === that-modality`) | 100% sampled | Section 1b failure mode #3 — these are the rows where the model SHOULD have skipped naming the absent modality |
| **Creator-flagged disagreement rows** (when a creator clicks "this score is wrong" in M2 result-card UI) | 100% sampled | The strongest single signal of rubric drift — user-behavior-vs-metric divergence per the framework's "signal-metric divergence" early-warning rule |
| **Rationale-inconsistency flag rows** (G7 fired) | 100% sampled for 7d, then drop to 10% | Section 1b failure mode #8 root-cause investigation |
| **Temporal-drift flag rows** (D16 fired) | 100% sampled | Section 1b failure mode #9 — every hit gets a Sentry issue with the rationale text + offending timestamp regex match |
| **Short-video rows** (duration ≤ 8s AND `gemini_body=false`) | 10% sampled monthly | D14 baseline — should be silent; sampling validates the body-skip path remains correct |

Human review of sampled rows is **monthly** for low-priority filters (10% sampling rows above) and **weekly** for the 100%-sampled critical filters. Each review session produces a `weekly_review.md` artifact under `.planning/benchmarks/phase5/reviews/<ISO-date>.md` that either confirms current rubric correctness OR opens a rubric-revision ticket against the next phase planning cycle.
