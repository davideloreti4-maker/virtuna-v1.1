Create `scripts/smoke-tiktok-pipeline.ts`. Two-mode runner:

**Mode 1 (default — UI-driven, recommended per D-25):** Read URL list, print instructions for the user to manually upload each video via the dev-server UI, then poll the latest `analysis_results` Supabase row to capture the prediction.

**Mode 2 (--direct):** Bypass the UI; POST directly to `http://localhost:3000/api/analyze` with `{ input_mode: "tiktok_url", tiktok_url: "..." }` for each URL in the list. Captures the SSE stream events + the final result.

Use Mode 2 for non-interactive automation; Mode 1 for the actual user-driven cadence (CONTEXT D-25: "user downloads TikTok video locally → uploads via Virtuna UI"). This plan's Task 5.2 uses Mode 1; Plan 06/07 can use whichever.

Structure:
1. Imports + env load (Pattern S8).
2. Argv parsing: `pnpm tsx scripts/smoke-tiktok-pipeline.ts <urls.txt> [--direct]`.
3. Read URLs from file (skip lines starting with `#` and blank lines).
4. For each URL:
   - Mode 1: Print `[video-N/N] Upload this URL via the dev-server UI: <url>`, wait for user keypress (use `process.stdin` resume), then query `analysis_results` for the latest row matching this URL or content_hash.
   - Mode 2: POST to `/api/analyze` with `tiktok_url` mode (per D-31 — this plan's 1 video can be tiktok_url to exercise D-31 verification). Capture SSE stream events; await the `prediction_complete` event; parse the final PredictionResult.
5. Capture per-video output:
   - Full PredictionResult (overall_score, confidence, factors, hook_decomposition, audio_signals, matched_trends, persona_simulation_results, platform_fit, counterfactuals)
   - Stage timings (from SSE stage events)
   - Total cost_cents
   - Signal availability (chip states)
6. Signal-completeness gate per CONTEXT (RESEARCH §SC#4 + 13-VALIDATION.md row 05):
   - Audio fingerprint non-null
   - Wave 3: ≥ 7 personas with verdicts
   - Wave 4: numeric platform_fit for at least TikTok
   - Stage 11: non-null CounterfactualResult with ≥ 1 suggestion
7. Write JSON output to `.planning/phases/13-real-pipeline-validation-production-hardening/validations/video-NN-raw.json` for each video (machine-readable input to the validation markdown).
8. Exit code: 0 if all videos pass the signal-completeness gate; non-zero otherwise.

9. **`--json-out <path>` flag (required for Nyquist sampling — Plan 05/06/07 verify probes):** when supplied, write the final aggregate result as JSON to the given path containing AT MINIMUM: `audio.embedding` (non-null when audio passed), `personas` (array; length >= 7 when Wave 3 passed), `platform_fit` (numeric), `counterfactuals` (object with `band` + `suggestions`), `cost_cents` (number). This is the structural probe contract downstream verify blocks assert against.
