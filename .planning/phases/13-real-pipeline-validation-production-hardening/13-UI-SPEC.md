## Surface 1: SuggestionsSection Rebuild

### Props Contract

```ts
// Each item from result.counterfactuals.suggestions[]
interface SuggestionItem {
  type: 'fix' | 'stretch' | 'reinforcement';
  headline: string;           // 1-line summary, ≤80 chars
  detail: string;             // Expanded explanation, 1-3 sentences
  timestamp_ms?: number;      // Optional video timestamp anchor
  signal_anchor: string;      // Which signal grounds this (e.g. "hook_score", "persona_dissent")
}

// Adaptive band shape per D-05
interface SuggestionsBandProps {
  band: 'low' | 'mid' | 'high';  // Derived from overall_score: <50 = low, 50-70 = mid, ≥70 = high
  suggestions: SuggestionItem[];
}
```

### Section Header — Adaptive per Band

| Band | Score range | Section header copy | Framing |
|------|-------------|--------------------|----|
| `low` | < 50 | "What to Fix" | 3 hyper-specific fixes |
| `mid` | 50–69 | "Improvements + What's Working" | 2 fixes + 1 reinforcement |
| `high` | ≥ 70 | "What's Working" | 1 stretch + 2–3 reinforcements |

Header rendered as `Text as="span" size="sm" muted` — matches existing section header pattern.

### Item Layout

```
┌─────────────────────────────────────────────┐
│  [type badge]   [signal_anchor text]  [⏱ 0:03]  │  ← row 1: badge + anchor + optional timestamp
│  Headline text at text-sm font-semibold      │  ← row 2: headline
│  Detail text at text-sm text-foreground-secondary │  ← row 3: detail
└─────────────────────────────────────────────┘
```

- Items separated by `border-b border-border` with `py-3`. Last item: `last:border-b-0 last:pb-0`.
- `type` badge: size="sm" (`text-[10px]`, `h-5`). Variant per type mapping above.
- `signal_anchor`: `Caption` component, uppercase tracking-wider, muted — identifies which signal grounds the suggestion.
- `timestamp_ms`: only renders when present. Format: `⏱ M:SS`. Use `Caption` muted. A clock icon (`Clock` from lucide-react, `h-3 w-3`) + formatted time string.
- `headline`: `text-sm font-semibold text-foreground`. One line, no truncation.
- `detail`: `text-sm text-foreground-secondary leading-relaxed`. 1-3 sentences.

### Empty / Degraded State

Per D-04: Stage 11 always runs — `never empty`. Section must never render null.
However, if `suggestions.length === 0` arrives despite D-04 (degraded backend path):

- Render section with header and single fallback item:
  - `headline`: "Analysis in progress"
  - `detail`: "Suggestions were not available for this prediction. Run another analysis to get actionable feedback."
  - `type`: `"reinforcement"` (neutral green — not alarming)
  - No `signal_anchor`, no `timestamp_ms`

### Always-On Contract

Remove the `if (suggestions.length === 0) return null` guard from existing `SuggestionsSection`. Replace with degraded state above.

## Surface 2: SignalAvailability Chip Three-State

### New Signal State Type

```ts
type SignalState = 'available' | 'disabled' | 'failed';

// Replace boolean in SignalAvailability with discriminated value
// The chip component maps SignalState → Badge variant + icon + opacity
```

### Chip Rendering Rules

| `SignalState` | Badge variant | Text | `className` |
|--------------|---------------|------|-------------|
| `'available'` | `success` | `Audio ✓` | `''` (no modification) |
| `'disabled'` | `default` | `ML ✕` | `'line-through opacity-40'` (carries Phase 11 pattern) |
| `'failed'` | `warning` | `Audio ⚠` | `''` (no modification — warning color is signal enough) |

### Updated CHIP_SIGNALS

Show 4 signals, same as Phase 11. Order: Audio, Personas, Retrieval, ML.

- `audio`: maps `signal_availability.audio` → state
- `personas`: maps `signal_availability.personas` → state  
- `retrieval`: maps `signal_availability.retrieval` → state (will be `'disabled'` per D-15)
- `ml`: maps `signal_availability.ml_classifier` → state (will be `'disabled'` per Phase 10)

### State Derivation

The source of truth is `signal_availability` on `PredictionResult`. The chip component receives either:
- `boolean` (backwards-compat for existing data) → treat `true` as `'available'`, `false` as `'disabled'`
- `SignalState` string when extended backend lands

Chip component must handle both shapes via a normalizer helper.

### Chip Row Layout

Unchanged from Phase 11: `flex flex-wrap gap-1.5`. All 4 chips always render — no conditional hide.

## Interaction Contract

### SuggestionsSection

- Static display only — no expand/collapse, no hover actions on items.
- No user interaction beyond scroll. Matches existing `SuggestionsSection` interaction model.
- Timestamp anchor (`⏱ 0:03`) is display-only in Phase 13. No seek-to behavior (M2 — requires video player wiring).

### SignalAvailabilityChips

- Static display only — chips are non-interactive (`Badge` component is display-only per `badge.tsx` docstring).
- No tooltips on chips in Phase 13 (tooltip on `'failed'` explaining the error is M2).
- All 4 chips always visible regardless of state.

## Degraded-Mode Rendering (D-12 text-only / script mode)

When `input_mode === 'text-only'` (all visual signals unavailable):

| Surface | Behavior |
|---------|----------|
| SignalAvailabilityChips | Audio `✕`, Personas `✓` (text can run personas), Retrieval `✕`, ML `✕`. All disabled chips use `variant="default" line-through opacity-40`. |
| SuggestionsSection | Still renders — Stage 11 always runs. Suggestions will be text-grounded. Section header adapts per score band normally. |

No special UI mode flag needed — the chip states communicate degradation naturally.
