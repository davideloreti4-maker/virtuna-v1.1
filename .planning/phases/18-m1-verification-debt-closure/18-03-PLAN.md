    ```typescript
    // IN-03 (VERIF-04 / T-06-13): SSRF guard before any fetch(sound_url).
    // Permissive by design — any public HTTPS hostname is allowed (users test
    // against TikTok CDN, IG, camera-roll exports, custom URLs). The guard
    // only rejects non-HTTPS schemes and private / loopback / link-local IPs.
    // On violation: log.warn + bare return (non-fatal — cron continues).
    if (row.sound_url) {
      let hostname: string;
      try {
        hostname = new URL(row.sound_url).hostname;
      } catch {
        log.warn("sound_url SSRF guard rejected — invalid URL", { sound_url: row.sound_url });
        return;
      }
      const isPrivate =
        /^localhost$/i.test(hostname) ||
        /^127\.\d+\.\d+\.\d+$/.test(hostname) ||
        /^10\.\d+\.\d+\.\d+$/.test(hostname) ||
        /^172\.(1[6-9]|2\d|3[01])\.\d+\.\d+$/.test(hostname) ||
        /^192\.168\.\d+\.\d+$/.test(hostname) ||
        /^169\.254\.\d+\.\d+$/.test(hostname) ||
        /^::1$/.test(hostname) ||
        /^f[cd][0-9a-f]{2}:/i.test(hostname);
      const isHttps = row.sound_url.startsWith("https://");
      if (!isHttps || isPrivate) {
        log.warn("sound_url SSRF guard rejected", { sound_url: row.sound_url });
        return;
      }
    }
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user-controlled URL → server-side fetch | `sound_url` originates from external scrapes (Apify) and is stored in `trending_sounds`. The cron then fetches it from the Vercel server. Without a guard, an attacker could store an internal URL (e.g. `http://169.254.169.254/...` for cloud metadata, or `http://10.0.0.1/admin`) and the server would issue a request to it. This is the classic Server-Side Request Forgery pattern. |
| Phase 12 reference: T-06-13 | This is the explicit threat-model item the milestone roadmap traces here. Closing IN-03 closes T-06-13. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-06 (closes T-06-13) | Information Disclosure | `processSoundEmbedding` server-side fetch | mitigate | Reject non-HTTPS schemes + RFC1918 + loopback + link-local + IPv6 ULA hostnames before any fetch. Implemented via `new URL(...).hostname` + 7 string regexes covering: 10/8, 172.16-31/12, 192.168/16, 127/8, 169.254/16 (link-local incl. AWS/GCP/Azure metadata), ::1, fc00::/7. HTTPS-only requirement also blocks `file://` and `gopher://` exploitation. |
| T-18-07 | Denial of Service | URL parsing throws on malformed input | mitigate | `try { new URL(...) } catch { log.warn; return; }` — malformed URL is rejected gracefully, cron continues. No throw bubbles out. |
| T-18-08 | Information Disclosure | DNS rebinding (hostname resolves to public IP at check, internal IP at fetch) | accept | Per D-12 / D-14 / Claude's Discretion section of CONTEXT.md: string-based regex check is sufficient for T-06-13 (which targets internal probing). DNS-rebinding is a known higher-effort attack class; the milestone explicitly accepts this risk in exchange for keeping the guard permissive for legitimate camera-roll / CDN URLs. Future hardening (e.g. actual DNS resolve + re-check at fetch time) is recorded under "Deferred Ideas" in CONTEXT.md. |
| T-18-09 | Tampering | Bypass via IPv4-mapped IPv6 (e.g. `::ffff:10.0.0.1`) | accept | Out of scope for permissive-by-design guard. Hostname extracted via `new URL(...).hostname` — if user-supplied URL uses IPv4-mapped IPv6 to reach internal range, regex misses. Acceptable for current threat profile (Apify-sourced URLs are public CDN domains, not IPv6 literals). Documented as residual risk. |
| T-18-10 | Repudiation | Silent SSRF rejection | mitigate | `log.warn("sound_url SSRF guard rejected", { sound_url })` records every rejection with the offending URL. Audit trail preserved. |
</threat_model>
