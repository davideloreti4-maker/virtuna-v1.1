## Color

Transcribed from `globals.css` `@theme`. **Monochrome by default. Accent is near-zero / liveness-only.**

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#262624` (`--color-background`, warm charcoal) | App background, home empty-state surface |
| Secondary (30%) | `#1e1d1b` composer (`--charcoal-composer`) · `#1a1a18` sidebar · `#2f2e2b` chip/lifted | Composer surface, picker popover, chips, hover-lifted rows |
| Accent (10% — effectively ~0% here) | `#d97757` terracotta (`--color-accent`) | See reserved list below — at MOST one element per view, often zero |
| Destructive | `--color-error` | Not used in this phase (no destructive actions) — accent ≠ error, never conflated |
| Primary text | `#ece7de` (`--color-foreground`, cream — NEVER `#fff`) | Audience names, chip labels, skill labels |
| Secondary text | `#c2bdb4` (`--color-foreground-secondary`) | Row sub-meta |
| Muted text | `#8a857c` (`--color-foreground-muted`) | Section headers, trust badges, dismiss link, placeholders |
| Primary action | `#ece7de` cream (`--color-action`) | Build CTA / demo CTA / send — NEUTRAL cream, NEVER accent |
| Borders | `rgba(255,255,255,0.06)` rest, `rgba(255,255,255,0.1)` hover | Picker rows, chips, chooser, card edges |

Accent (`#d97757`) reserved EXCLUSIVELY for: **the single audience liveness/presence dot** (the "personas ready" / live-reactor signal in `audience-presence.tsx`) and the SIM/constellation lit node. NOTHING ELSE on this surface gets accent — not the 3 chips, not `+ Build an audience`, not selected/active picker rows, not skill pills, not the demo CTA, not trust badges, not the Build chooser paths. Selection/active/focus state → use cream weight + `#2f2e2b` lifted background + 10% border, never paint. Matte only: no glass gradient, no glow, no white inset-shine, no `backdrop-filter` class (per the known Lightning-CSS issue, blur goes via React inline `style` if ever needed — but nothing here needs it).

## Surface Contracts (the phase-specific design + interaction lock)

> These five surfaces ARE the phase. Each states the Socials/byte-identical render first, then the additive General branch.

### S1 — In-composer Audience picker (UX-01 / D-02)
**Enhance the LIVE `AudiencePresence` switcher** (`audience-presence.tsx` L307–359). Do NOT resurrect `audience-chip.tsx` (dead; markup reference only — resurrecting doubles the picker).

- **Anchor:** unchanged — the existing switcher popover in the composer left control cluster, opening upward, Radix-popover behavior (Escape / outside-click) reused as-is.
- **Rows:** replace the flat `audiences.map(...)` with `groupAudiences(audiences)`. Render under section headers:
  - `── Socials ──`: `baseline + templates + yours.filter(mode==='socials')`
  - `── General ──`: `generalTemplates + yours.filter(mode==='general')` — **rendered only when non-empty**.
- **Section-header style:** 12px / 500 / muted (#8a857c), uppercase, with a hairline 6% rule on each side (the `──` divider motif). 8px gap above/below. It is a label, not a selectable row.
- **Row layout:** `[name (14/500/cream)] ............ [trust badge (11/500/muted, right-aligned)]`. Row padding 8px vertical / 12px horizontal, radius 8px, hover → `#2f2e2b` + 10% border. Selected row → cream weight + lifted bg, NO accent.
- **Trust badge:** `resolveTier(audience)` → `Directional` (all General) / `Validated`. Neutral muted text only. Import the leaf tier helper, never the pack barrel (BUILD-01 bundle-leak discipline).
- **`+ Build an audience`:** a final full-width row below all sections, separated by a 6% hairline. Cream label, plain `+` glyph (Phosphor), NO accent. Opens the S3 chooser.
- **Byte-identical guard:** a creator with only a Socials/personal audience sees ONLY the `── Socials ──` section (or no header at all if the existing single-section render is what ships today — match current default), default-selected row unchanged, `+ Build an audience` appended. No General header for them.

### S2 — Mode-scoped skill menu (UX-02 / D-01 / D-07)
`composer-controls.tsx` `SKILLS` / `SkillRows`. Add `modes: ('socials'|'general')[]` to `SkillMeta`; filter `SKILLS` by `modes.includes(activeMode ?? 'socials')` BEFORE the existing group partition. Thread `activeMode = selectedAudience?.mode ?? 'socials'` from `composer.tsx` → `ComposerControls` → `SkillRows`.

- **Socials render (BYTE-IDENTICAL — #1 constraint):** when `activeMode` is absent or `'socials'`, the menu shows the EXACT current creator skills (Hooks / Test / Remix / Chat …) in the EXACT current order, with the existing Creator/Marketing sub-group headers unchanged. No reorder, no relabel, no restyle. Default when no audience selected = `'socials'`.
- **General render (ADDITIVE):** when `activeMode === 'general'`, show **Profile / Simulate / Predict** (in that order). Same row visual language as today's skill rows (14/400/cream, 8px radius hover-lift, NO accent on pills — dosage rule).
- **Per-skill submit semantics (D-07 — the central gap):**
  - **Profile** → does NOT submit the topic field. Focuses/opens the existing evidence-drop affordance (`Attach a chat or screenshot`, composer.tsx ~L1596) with a General-appropriate placeholder.
  - **Simulate** → requires a selected General audience; field becomes "type a draft message" → existing `POST /api/tools/simulate` with `audienceId = selectedAudience.id`.
  - **Predict** → requires a selected General audience/panel + scenario → existing `POST /api/tools/predict`.
  - Gate (§16.4 asymmetry): Simulate/Predict require a General audience first; if none selected, the row routes the user to Build. Profile is the door that creates one.
- **No new route, no engine call.** ENGINE_VERSION stays `3.20.0`.

### S3 — Build-an-audience chooser (UX-04 / D-03 / D-08)
**Recommendation: a small centered Radix `Dialog` modal** (radius 12px, `--charcoal-composer` surface, 6% border, `--shadow-float`), NOT an inline composer step — the three paths diverge (one hops away) so a modal reads as a deliberate one-time setup action and keeps the slim composer uncluttered.

- **Layout:** title `Build an audience` (14/500/cream) + three stacked path rows, 16px gap, each a full-width selectable row (8px radius, hover-lift, NO accent):
  1. `From a description` → on select, HOP to `/audience/new` with `mode:'general'` preset; on save, return to composer with the new SIM selected (D-08).
  2. `From evidence` → stays in-composer; routes to P5 `profile-runner` / evidence drop; on bake, saved General SIM auto-selected.
  3. `From a template` → stays in-composer; lists `GENERAL_TEMPLATES` (analyst / hiring); on pick → clone-and-edit: strip sentinel id, `createAudience({ ...cloneable, name: derivedEditableName, mode:'general' })`, then open the name field pre-filled + editable.
- **Auto-naming:** each path pre-fills an editable name (see Copywriting). Never insert a `SENTINEL_ID`.
- **Each path label is plain cream** — no accent, no colored icons. Phosphor glyphs at muted weight if used.

### S4 — Home empty state + first-run demo (UX-05 / D-04)
`home/page.tsx` (unlock the LOCKED-empty block) + `home-greeting.tsx` + `home-page-layout.tsx`.

- **Vertical rhythm:** greeting (serif voice-line, unchanged) → composer → 3 chips, with `lg` (24px) between blocks, `xl` (32px) above the chip row.
- **Chips:** the 3 LOCKED-verbatim chips in a single horizontal row (wrap on narrow), `md` (16px) gap. Each chip: 14/400/cream, `#2f2e2b` lifted surface, 8px radius, 6% border (10% hover), 8px×12px padding. **NO accent.** Tapping a chip pre-fills the composer / opens the relevant flow (Test → idea submit on the active audience; Profile → evidence drop; Predict → Predict flow gated on a General audience).
- **First-run demo:** a one-tap, show-once, dismissable affordance beneath the chips. CTA `See it in action` (neutral cream button, `--color-action`). On tap → runs the pre-seeded sample chat through Profile→Read (existing `POST /api/tools/profile` with a canned fixture) — the cold-start wow, no chat-export friction. `Dismiss` text link (12/400/muted) to the right.
- **Show-once persistence:** `localStorage` flag (in-repo precedent: `use-lens-scale.ts`, `society-selector.tsx`). Set on first run OR on dismiss; thereafter home returns to plain greeting + composer + chips. Single-device is acceptable (D-04) — no DB/profile field.
- **Byte-identical note:** the greeting + composer themselves are unchanged; chips + demo are an ADDITIVE block in the previously-empty region.

### S5 — Generalized ambient reactor (UX-03 / D-06)
`audience-presence.tsx` reactor view. **Mechanism UNCHANGED** — reuse the existing focus/explicit-ask reactor (`AbortController` throttle); NOT a new per-keystroke loop. Only the VISUAL delta for General is specified:

- **Socials (BYTE-IDENTICAL):** the existing multi-persona presence (dots / panel) renders exactly as today via `buildAudienceRepaint`. Do NOT touch the `audienceRepaint === undefined` no-op branch (the byte-identical engine guard).
- **General — multi-persona panel SIM:** same multi-persona presentation as Socials (the panel reactor), driven by "the active Audience."
- **General — person SIM (1 persona):** present as a **single reactor** — one identity reacting *as that one person* (single dot/avatar + single reaction stream), not a swarm of dots. This is the only visual divergence.
- **Liveness dot:** this is the ONE sanctioned accent (`#d97757`) on the surface — the single presence/"reacting" signal. At most one, often the only color on screen.
- Cost stays bounded by the existing Socials debounce/throttle. Creator behavior byte-identical.

## Byte-Identical Contract (the #1 constraint — RESEARCH Pitfall 2)

For every shared surface, the Socials/creator-default render is UNCHANGED from today; General is an additive branch that appears only when a General audience is active/owned.

| Shared surface | Socials render (UNCHANGED) | Additive General branch |
|----------------|----------------------------|-------------------------|
| Skill menu (`SkillRows`) | Hooks/Test/Remix/Chat in exact current order + Creator/Marketing sub-headers; default `activeMode='socials'` | Profile/Simulate/Predict only when `mode==='general'` |
| Picker switcher (`AudiencePresence`) | Default-selected personal/baseline audience; no `── General ──` header when none owned | `── Socials ──`/`── General ──` sections + `+ Build an audience` |
| Home layout | Greeting + composer unchanged | 3 chips + show-once demo in the previously-empty block |
| Reactor (`AudiencePresence`/`buildAudienceRepaint`) | Multi-persona dots; `undefined`→no-op branch untouched | Single-reactor framing for person-SIM |

**Guard tests (must stay green / be updated, never weakened):**
- `composer-controls.test.tsx` — UPDATE: assert mode-scoped render (Socials = creator skills unchanged + no-audience default = Socials; General = Profile/Simulate/Predict).
- `home.test.tsx` — UPDATE: assert the 3 locked chips + one-tap demo + show-once (replaces the now-obsolete NO-chips/NO-demo assertions).
- `audience-presence.test.tsx` — EXTEND: assert mode sections + `+ Build an audience` + General drives the reactor.
- `audience-regression-gate.test.ts` — KEEP GREEN (BLOCKING): ENGINE_VERSION `3.20.0`, General injects no override.
- `reskin-matte.test.ts` — KEEP GREEN: no coral/glass regression on transplanted surfaces.
