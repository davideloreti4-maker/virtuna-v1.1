<objective>
Thread verbatim through the remaining hops (assembly to aggregator to PredictionResult to DB) following the emotion_arc precedent, on BOTH axes — the hook AND per-segment — persist the whole payload to a dedicated verbatim JSONB column (Option A), and bump the cache key. After this plan the full pipeline carries verbatim (hook + segments) end-to-end and a real run populates the column; Plan 03 proves it.

**BLOCKER fix (per-segment persistence path):** per-segment spoken_text/on_screen_text is not just plucked off the hook — it rides on the normalized segments. This plan (1) preserves those fields through normalizeSegments so they reach pipelineResult.segments, (2) extends VerbatimPayload to { hook?, segments?[] }, (3) derives verbatim.segments from the normalized segments in the aggregator pluck (each entry { idx, spoken_text, on_screen_text }, null preserved — D-02 vs the [inaudible] marker D-04.2), and (4) persists the WHOLE verbatim object (hook + segments) at both route sites in the same verbatim JSONB column — no second column. D-03.2: per-segment text is what P4 Audience-Sim needs to know what viewers heard at each point.

Persistence decision (Claude's Discretion, locked to Option A): one dedicated verbatim JSONB column on analysis_results, mirroring emotion_arc's migration 20260531000000. Verbatim is produced inline by the aggregator like emotion_arc, so it rides the SAME row write with zero new write-race surface; R1's "persists non-empty verbatim" is query-clean (hook at verbatim->'hook', segments at verbatim->'segments'); verbatim is first-class P3/P4 sensor data. Minimal migration (one ADD COLUMN IF NOT EXISTS, no backfill).

Additive-only (D-01): every edit ADDS. No score-blend input removed/repointed.
