<objective>
**Wave 0 Smoke Test — Gemini 2.5 Flash Audio Reliability + HOOK-02 Traceability Migration.**

Purpose: This plan ships two cheap, parallel Wave 1 deliverables that share no file conflicts:

1. **Gemini audio reliability smoke test (load-bearing).** Phase 6's load-bearing assumption (D-A1 + Assumption A1 in RESEARCH) is that Gemini reliably emits voice_clarity_0_10, audio_hook_first_2s_0_10, three ratios summing to ~1.0, and a 50-150 char audio_description when the existing `gemini_video_analysis` prompt is extended with an `audio_signals` schema block. If reliability is poor (<90% emission rate or malformed fields), `signal_availability.audio = false` would fire often and the entire audio signal becomes near-useless in production.

2. **HOOK-02 REQUIREMENTS.md traceability migration (per D-H1).** D-H1 mandates moving HOOK-02 from Phase 5's "Multi-Modal Hook Decomposition" section into Phase 6's "Audio Analysis" section. ROADMAP already records the migration; REQUIREMENTS.md still places HOOK-02 under the old section. This plan finalizes the migration.

This plan creates a standalone smoke-test script the developer runs ONCE against 3 fixture videos (talking_head, slideshow, music-heavy) BEFORE writing any production audio code. The output is a human-reviewable JSON dump showing what Gemini actually emits for each video. The developer signs off (or aborts the phase) based on the results. Plan 03 lands the real schema extension; this plan only validates that the extension WILL work.

This plan gates Phase 6 SC#1 — if Gemini reliability is poor, Plans 02-06 should not ship as written; the planner returns to discuss-phase to revisit D-A1.

Output: A `scripts/smoke-test-gemini-audio.ts` script (tsx-runnable), a fixture-directory README, and a finalized REQUIREMENTS.md that owns HOOK-02 in the Audio Analysis section.
