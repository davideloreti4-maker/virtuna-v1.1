## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local CLI → Supabase project | `supabase db push` authenticated via CLI session or SUPABASE_ACCESS_TOKEN |
| Migration → existing analysis_results data | ALTER TABLE on a table with existing rows |
| Regenerated types → application code | TypeScript types from live DB schema |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-22 | Tampering | Migration applied to wrong Supabase project | mitigate | Task 2 [BLOCKING] checkpoint REQUIRES human to verify `supabase projects list` shows correct project before push. Migration is IF NOT EXISTS so re-runs are safe. |
| T-03-23 | Denial of Service | ALTER TABLE blocks reads/writes on a large analysis_results table | accept | analysis_results is small (likely <100k rows in dev/early-prod); ADD COLUMN IF NOT EXISTS with DEFAULT '{}' on JSONB is fast in PostgreSQL 11+ (no full table rewrite). CREATE INDEX IF NOT EXISTS uses BUILD NOT CONCURRENTLY by default — for prod with high-traffic, could be CONCURRENTLY. Accept for dev. |
| T-03-24 | Information Disclosure | Regenerated types leak schema secrets to git history | accept | database.types.ts is generated metadata — no secrets, just column names and types. Already committed in prior phases. |
| T-03-25 | Tampering | Migration timestamp collision with concurrent unmerged branch | mitigate | Task 1 explicitly checks `ls supabase/migrations/` for latest timestamp before creating new file. If a collision arises post-merge, supabase CLI surfaces the conflict — operator resolves. |
