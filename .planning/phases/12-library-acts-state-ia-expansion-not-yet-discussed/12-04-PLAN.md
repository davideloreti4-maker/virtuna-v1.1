<objective>
Ship persona editing (AUD-EDIT-01 / D-06): edit a calibrated audience's personas — **Name · Disposition · Temperature · Description** — written to the **per-audience override slot** (the audience's `personas` JSONB via the shipped `PATCH /api/audiences/[id]`), **never** the General baseline (read-only; the regression gate protects it). This mirrors the P10 D-05 recalibration write pattern and is gate-safe by construction: General is a virtual constant with no DB row (`personas: []`) and no Edit affordance, so it is structurally unwritable. Direct weight/share/distribution editing is REJECTED (gate risk, out of scope).

Name needs a real persistence target: `CalibratedPersona` currently has `{ archetype, repaint, temperature, disposition, share }` and NO name/label field (the display name is derived everywhere from `archetype.replace(/_/g," ")…`). D-06 lists **name** as an editable field, so this plan adds a presentation-only `label?: string` to `CalibratedPersona` — defaulting to the archetype-derived string, written via the `personas` JSONB override — **explicitly outside the regression-gate surface**: all five runners (`ideas/hooks/script/remix-runner.ts`, `two-audience-read.ts`) build their repaint map from `[p.archetype, p.repaint]` ONLY and never read `label`, and General has no DB row, so adding `label` cannot move any SIM output.

Purpose: persona editing lets the creator tune who Numen writes and tests for, on their own calibrated audiences, without ever touching the protected baseline — now unblocked because P8 W0 tuned the persona values P7 deferred on.
Output: a presentation-only `label?: string` on `CalibratedPersona`, a `persona-edit-form.tsx`, a per-persona Edit affordance + `label ?? archetype-derived` display + D-06 read-only-General caption in `audience-profile-view.tsx`, a unit test, and a regression-gate verification that the General baseline + ENGINE_VERSION are untouched.
</objective>
