# Quick Task 260607-00u: Build "Ask the expert" chat dock — Summary

Zero-engine-cost conversational surface that makes the Apollo expert interrogable after analysis completes. Grounded entirely on the cached `analysis_results` row. Streams Qwen answers token-by-token via the existing DashScope client.

## What Was Built

### Task 1 — DB migration + context builders (commit `08e622ee`)

`analysis_chats` table persists per-analysis conversation turns. Schema:
- `id uuid`, `analysis_id text` (FK → analysis_results, ON DELETE CASCADE), `user_id uuid`, `role text check ('user'|'assistant')`, `content text`, `scope text`, `created_at timestamptz`
- RLS: SELECT + INSERT policies, ownership-via-join (same pattern as 20260526000000), `user_id = auth.uid()` on both
- Index `analysis_chats_analysis_id_created_at_idx` for ordered history reads
- Applied to remote Supabase project `qyxvxleheckijapurisj`

`buildChatSystemContext(row, scope)` — frames Qwen as Apollo expert; DUAL-READ (`variants?.apollo ?? apollo_reasoning`); scope-filtered (audience→heatmap+behavioral, verdict→score+capper, engine/actions→dimensions, content→rewrites+text, null→whole-analysis). Explicitly instructs model to answer only from provided data, never mention external model names.

`deriveSeedPrompts(row)` — derives 2-3 questions: ceiling_capper → "Why will this flop?", weakest dimension → "How do I improve my X?", hook rewrite → "Can you give me a better hook?". Falls back to 2 generic prompts for bare rows.

16/16 tests GREEN.

### Task 2 — Streaming chat API route (commit `7fe48e72`)

`GET /api/analyze/[id]/chat` — returns ordered history (RLS enforces ownership).

`POST /api/analyze/[id]/chat`:
- Auth (401), message validation (400 on empty / >2000 chars), scope allowlist
- IDOR guard: `.eq('id', id).eq('user_id', user.id)` before any data leaves (T-00u-01)
- Per-analysis cap check (20 user turns → 429) + per-user rate limit (10 msgs/60s → 429)
- Builds messages: system context + prior turns + new user message
- Inserts user turn before streaming (so it's persisted even if client disconnects)
- Streams `event: token { delta }` frames from `QWEN_REASONING_MODEL` at `temperature: 0.3`
- On stream complete: inserts assistant turn, emits `event: done { content }`
- Qwen/DashScope only — no Claude/Gemini imports anywhere

10/10 tests GREEN; build passes.

### Task 3 — Dual-mode UI (commit `d74a5c75`)

`useExpertChat(analysisId)` — SSE POST body-reader hook mirroring use-analysis-stream pattern. Manages `messages`, `streamingText`, `isStreaming`, `error`. `loadHistory()` called on mount for permalink replay. Optimistic user turn appended immediately on send; rolled back on error.

`ExpertChatThread` — fixed DOM overlay (`z-[199]`, positioned above bar via `bottomOffset`), `max-height: min(60vh, ...)`, `overflow-y-auto`, auto-scrolls to newest. Raycast glass gradient + `border-white/[0.06]` + inset shadow. Mobile: full-height via CSS constraints.

`ExpertChatInput` — seeded prompt chips (click → instant send), removable "about: frame" scope chip from `board-store.selectedNodeId` (mapped to frame label + scope key), textarea + submit, demoted "+ New analysis" that calls `triggerNewAnalysis + openInputDrawer`.

`CommandBar` extended with `completedAnalysisId` + `analysisRow` props. Pre-analysis path: unchanged ContentForm (byte-identical render). Post-analysis: ExpertChatInput + ExpertChatThread. Thread rendered outside the bar container (no layout impact on bar). Board.tsx passes the props when `boardMachineState === 'complete' | 'anti-virality'`.

26/26 chat tests GREEN; build + lint clean.
