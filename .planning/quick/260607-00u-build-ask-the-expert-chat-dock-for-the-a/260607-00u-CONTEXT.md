# Quick Task 260607-00u: Build "Ask the expert" chat dock for the analysis board - Context

**Gathered:** 2026-06-06
**Status:** Ready for planning — all gray areas resolved in brainstorm (decisions LOCKED, do not revisit)

<domain>
## Task Boundary

Add a conversational "Ask the expert" surface to the analysis board so a user can
interrogate the cached analysis insight after an analysis completes. The chat is the
Apollo expert (the same interpreter that produced the InsightHero) made interrogable.

In scope: chat dock UI, Qwen-backed answer endpoint, persistence + permalink replay,
seeded prompts, scope chips, mobile sheet.

Out of scope (v1): coach/what-if re-runs, frame-citation camera-jump (deferred v2),
full per-user quota system.
</domain>

<decisions>
## Implementation Decisions (LOCKED)

### Purpose
- Explainer + expert. Interrogate the EXISTING cached analysis ("why will this flop?")
  + general expert advice.
- Grounded entirely on cached analysis data — ZERO new engine cost.
- NO what-if / coach / re-run scenarios in v1.

### Placement
- Extend the existing CommandBar (`src/components/command-bar/CommandBar.tsx`) into a
  dual-mode dock. DO NOT build a separate drawer or a Konva board frame.
- Pre-analysis: CommandBar = today's analyze form, UNTOUCHED.
- Post-analysis: primary action becomes "Ask the expert" input; "+ New analysis"
  demoted to a small secondary affordance.
- Conversation thread expands UPWARD from the bar as a FIXED DOM overlay (~60vh,
  internal scroll). Board stays visible behind it and never reflows.
- Chat MUST survive Konva pan/zoom — it is DOM overlay, NOT a board node/frame.

### Grounding / scope
- Seeded prompts: 2-3 highest-leverage questions derived from the analysis result,
  shown above the input so there is no blank-box cold start.
- Scope chips: board-store already tracks `selectedNodeId`. Selecting a frame injects
  a removable "about: <frame>" chip into the input. Default scope = whole analysis.

### Streaming
- STREAM answers token-by-token. Reuse the existing SSE pattern
  (`src/hooks/queries/use-analysis-stream.ts` / useAnalysisStream).

### Cost / rate guard
- Simple per-analysis message cap (e.g. ~20 messages/analysis) + basic per-user rate
  limit. No full quota system.

### Mobile
- Thread becomes a full-height sheet; reuse the existing CommandBar grabber pattern.

### Claude's Discretion
- Exact seeded-prompt copy + which 3 questions to derive (planner/executor picks from
  apollo_reasoning: e.g. ceiling_capper → "why will this flop", weakest dimension →
  "how do I fix X", top hook rewrite → "rewrite my hook").
- Component file naming/structure under src/components/command-bar or a new chat dir.
- Exact message-cap number and rate-limit window.
- analysis_chats table column types/indexes (within the shape below).
</decisions>

<specifics>
## Plumbing constraints (HARD)

- **Model: Qwen on DashScope ONLY.** Pipeline is Qwen-only — NO Claude / Gemini /
  DeepSeek in-product. The chat endpoint must call Qwen via the existing DashScope
  client used by the engine.
- **Context seed:** `apollo_reasoning` + `behavioral_predictions` + `heatmap` +
  verdict, filtered by the active scope chip. All already persisted in the
  `analysis_results` table (Supabase), keyed by analysis ID in the URL
  (`/analyze/[id]`).
- **Persistence:** new `analysis_chats` table — `analysis_id` FK, `role`
  (user|assistant), `content`, `scope`, `created_at`. Conversation must replay on
  permalink reload (read history by analysis_id on mount).
- **Streaming:** reuse the existing SSE pattern.

## Deferred to v2 (do NOT build now)
- Frame-citation camera-jump (answer cites a frame → clickable → camera pans via
  board-store presets). Design the answer rendering so this can be added later, but do
  not implement the citation-parsing / camera wiring in v1.
- Coach / what-if scenario re-runs.
</specifics>
