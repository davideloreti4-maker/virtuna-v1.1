## 2. üî¥ THE KEY LESSON ‚Äî a card field is hand-picked at THREE layers, not one

This is the trap the next session MUST know. Wiring a new card-prop end-to-end means editing **three**
places per skill ‚Äî miss any one and the field is silently dropped on the LIVE path (it only reappears
after a page reload, from the persisted block):

1. **Runner** (`src/lib/tools/runners/<skill>-runner.ts`) ‚Äî add to the output contract (the JSON
   `Shape:` string + a rule sentence), parse it, map it into `blockData.props`. Grounded/targeted
   runs have SEPARATE contract strings ‚Äî use shared fragments (see `FILMING_*` / `VISUAL_HOOK_*`
   consts) so it lands once. Remix generates in `src/lib/engine/remix/adapt.ts` (+ its zod schema +
   `decode-types.ts` AdaptConcept type).
2. **Route SSE `content` event** (`src/app/api/tools/<skill>/route.ts`, remix = `.../remix/run/route.ts`)
   ‚Äî the `send("content", { blocks: blocks.map(...) })` HAND-PICKS props field-by-field. Add yours.
3. **Client stream hook** (`src/hooks/queries/use-<skill>-stream.ts`) ‚Äî THREE sub-spots: the
   `PartialXCard` interface, the content-event mapper (re-builds the card field-by-field, incl. a
   per-BEAT re-map for script), and `toBlocks()`. Hooks stream has TWO content mappers.

The code comments already warn about this for `proof`/`target`/`population` ("same reload-only
hazard‚Ä¶ must ride the face, not just persist") ‚Äî the new fields follow the exact same pattern.

**The in-chat dispatch path is DIFFERENT and needs NO per-field work:** `skill-dispatch.ts` ‚Üí
chat route `send("block", { block })` ‚Üí `use-chat-stream.ts` all pass the block **whole**;
`MessageBlocks` schema-revalidates + renders. Verified clean this session. Only the DEDICATED
composer streams hand-pick (they reconstruct typed partial cards for the deferred score-patch).
